# Сводка решений по change `add-agent-executor`

> Временный файл — результаты explore-сессии по scope следующего change после `add-manual-run`: реальный `agent-cli` StageExecutor (Claude Code CLI через ProcessBuilder). Не для коммита. При propose — сырьё для proposal/design.

**Рамка (зафиксировано на входе):**
- Ценность: тот же `gnomish run`, но стадию выполняет реальный агент; человек остаётся на ролях из change 2 (эскалации, judge, external). Первая проверка гипотезы «AI-гном проходит QC-цикл».
- Вне scope: git/worktree, трекер, персистентность, бюджеты, песочница (ручной запуск в своём репо под присмотром — то же обоснование, что для command-чеков), ai-provider порт / api-executor.
- Порт готов: `StageExecutor.Request` (context, stage, workspace, attempt, feedback) → `ExecutionResult.Completed | DecisionNeeded`, оба с `ExecutorUsage` + `ToolTrace`. Схема `settings` — на усмотрение адаптера (map опаковый). Контрактная сьюта порта из change 2 — обязательна для нового адаптера.

## Темы

- [x] 1. Протокол DecisionNeeded (wire-контракт с агентом) — решено: decision-файл
- [x] 2. Ретраи: fresh run vs resume сессии — решено: fresh run
- [x] 3. Телеметрия: stream-json → ExecutorUsage/ToolTrace — решено полностью (токены, тайминги, live-прогресс)
- [x] 4. Judge: внутрь scope или наружу — решено полностью: внутрь, CliJudgeVoter; вердикт — финальное сообщение; телеметрия зеркалирует executor; wiring — манифест-driven + `--interactive`
- [x] 5. Тестовая стратегия — решено: 3 слоя (reference-дампы, фейковый агент, E2E = Ollama + ручной платный smoke)
- [x] 6. Схема settings — решено: два дома (манифест vs инсталляция), зашитая read-only политика судьи, строгая валидация ключей
- [x] 7. Общий «stage briefing builder» — решено: общий секционный рендер; чтение control-файла у адаптеров (человек — заглушка, агент — стоп); судья — подмножество без фидбека
- [x] 8. QC-петля глазами агента — решено: rework-преамбула, verify-план в промпте целиком, резюме раунда → RoundFinished + лог (доработка при git-workflow)

## 1. Протокол DecisionNeeded — РЕШЕНО: decision-файл

**Ключевая рамка:** протоколу не нужна пуленепробиваемость — QC-цикл является страховочной сеткой. Пропущенный сигнал (агент хотел спросить, но не распознан) → Completed → verify падает → quality failure → findings уходят фидбеком; цена — одна попытка. Ложный сигнал → эскалация без сжигания попытки; цена — минута внимания человека. Оба отказа деградируют мягко ⇒ критерий выбора — простота + портируемость, дефолт: **нет явного сигнала = Completed**.

**Механизм:** в промпт раунда добавляется инструкция «нужно решение человека — запиши JSON `{question, options[]}` в файл по пути `$GNOMISH_DECISION_FILE` и завершись». После выхода процесса: файл есть → `DecisionNeeded`, нет → `Completed`.

**Под-решения:**
- **Хранение:** временный каталог раунда, создаёт адаптер (`Files.createTempDirectory`), путь агенту через env/промпт. Вне workspace — ничего не просочится в будущую ветку задачи (git-workflow); путь уникален на раунд — протухший файл прошлой попытки невозможен по построению. Жизненный цикл владеет адаптер: создал → запустил агента → прочитал → удалил каталог.
- **Файл — транспорт на один хоп, не хранилище.** Вопрос сразу поднимается в домен: `ExecutionResult.DecisionNeeded` → `EscalationReport.DecisionNeeded`. Доставку человеку делают каналы эскалации других changes: консольный диалог (change 2), state-файл ветки → `gnomish status` (change 3 — требование «EscalationReport в state-файле» уже зафиксировано там), комментарий в трекере + Blocked (factory loop). Этому change для трекера делать ничего не нужно.
- **Толерантное чтение:** JSON невалиден → весь текст файла = `question`, options пустые (агент явно хотел спросить — потерять вопрос хуже, чем показать сырым); пустой файл → фолбэк-текст (доменный рекорд требует непустой question). Сырое содержимое файла при проблемном парсе логируется (WARN, полное содержимое, под `AttemptKey`) — *исправлено при обсуждении темы 3: в `ToolTrace` payload'у некуда лечь, диагностика — забота лога.*
- **Телеметрия** у DecisionNeeded та же, что у Completed (порт требует usage+trace у обоих вариантов) — собирается из stream-json независимо от исхода.
- **Риск:** запись вне workspace может требовать permission-настройки CLI (allowlist на конкретный путь) — решить в теме 6 (settings). Запасной план, если больно: путь внутри workspace (`.gnomish/runtime/`), адаптер удаляет после чтения.

**Отклонено:**
- *JSON в финальном сообщении агента* — отбирает канал человекочитаемого резюме раунда под машинный формат; хрупкий парс свободного текста (fences, приписки).
- *MCP-инструмент `request_decision`* — выигрыш только в схема-валидации трёх полей (остановить агента инструмент всё равно не может), цена — MCP-сервер, конфиг-инъекция, жизненный цикл. Пересмотреть, если практика покажет плохую комплаентность файлового протокола.
- *Exit-код агента* — CLI не даёт агенту управлять своим exit-кодом.
- Протокол — деталь **этого адаптера**, не порта: будущий api-executor решит иначе (native tool use), порт это уже скрывает.

## 2. Ретраи — РЕШЕНО: fresh run

Каждая попытка — новый запуск CLI (`claude -p`) в той же рабочей копии, фидбек прошлых попыток рендерится в промпт из `Request.feedback`. Почему:

- **Семантика порта уже такая:** `Request` самодостаточен (несёт полную историю фидбека), адаптер stateless между вызовами — resume завёл бы скрытое состояние (session-файлы CLI) вне контракта.
- **Стейтлесность инстансов:** session-файлы локальны для машины; после change 3 попытку N+1 может выполнять другой процесс/инстанс — resume дал бы разное поведение ретрая в зависимости от того, кто его выполняет. Конфликт не временный, а постоянный — дверь закрыта, не отложена.
- **Качество:** контекст провалившегося раунда содержит заблуждение, приведшее к провалу; свежий запуск с дистиллированными findings работает лучше, чем углубление в замусоренную сессию.
- **Непрерывность даёт workspace,** не сессия: стадийный контракт — та же рабочая копия для agent-cli, наработанные файлы на месте.

**Отклонено:** `--resume <sessionId>` — единственный плюс (экономия токенов на переисследование репо) ограничен `attemptLimit` сверху и съедается риском худшего результата; даже как внутрипроцессная оптимизация создаёт гетерогенность поведения.

Рендер фидбека в промпт — часть briefing builder (тема 7).

## 3. Телеметрия — токены РЕШЕНЫ; тайминги и live-прогресс обсуждаются

**Рамка: двойная роль stream-json.** Поток — не только телеметрия. Result-событие = финал раунда (существенное: провал его парса = infrastructure failure); телеметрия = best-effort (провал парса → `ExecutorUsage.none()` + пустой trace, раунд НЕ падает — та же мягкая деградация, что в теме 1). Неизвестные типы событий и лишние поля молча игнорируются: формат полудокументирован и дрейфует между версиями CLI.

**Токены — РЕШЕНО:**
- `TokenUsage` расширяется до 4 полей: `input`, `output`, `cacheCreation`, `cacheRead` — примитивные `long`, без per-field nullability («известны ли токены вообще» остаётся заботой контейнера). Мотив: для агентных нагрузок cache_read доминирует (в разы больше input по реальной стоимости); лосси-маппинг в 2 поля ошибается на порядок — в обе стороны, смотря как маппить.
- `ExecutorUsage.tokens: @Nullable TokenUsage` → `tokensByModel: Map<String, TokenUsage>`, **map-only** (без дублирующего тотала: точная избыточность = приглашение к рассинхрону). Пустая map = не зарепорчено (unknown ≠ zero сохраняется). Merge — union по ключам + `TokenUsage`-суммы (паттерн `mergeTools`). Тотал — производный хелпер для дисплея/status report.
- Ключи — **резолвленные model id** из `modelUsage` result-события (алиасы из конфига мутируют с правкой манифеста). Фолбэк для старых CLI без `modelUsage`: весь usage под главной моделью из `init`-события. Точную форму проверить на reference-файлах (тема 5).
- Обоснование per-model: раунд agent-cli мультимоделен (haiku на вспомогательных задачах, сабагенты); токены разных моделей не аддитивны; после отказа от costUsd идентичность модели — единственный путь к будущей стоимостной интерпретации (now-or-never: историю потом не расщепить).
- **Деньги: НЕ тащим.** `total_cost_usd` фиктивен при подписке (считается по API list price) и производен от токенов+модели. Денежные бюджеты выпилены из проекта: rules поправлены (traceability.md, proposal-format.md); в change остаётся хвост — `openspec/specs/pipeline-config/spec.md:97` (через дельту) и javadoc'и `AutonomyLimits`/`AutonomyDto`/`TaskState`/`AutonomyLimitsSpec`.
- Status report v1 (add-manual-run) идёт как есть; изменения схемы usage вносим поверх в этом change.
- `JudgeUsage` получает 4-польный `TokenUsage` прозрачно; вопрос «модель голоса» отложен к теме 4.

**Маппинг остального (лин, не зафиксировано):** `ToolTrace.calls` ← tool_use-события, `seq` = порядок чтения; `ExecutorUsage.tools` — агрегат, выводимый из trace; `wallTime` — адаптер сам мерит start→exit процесса (не зависит от парса result-события).

**Тайминги — РЕШЕНО:**
- События stream-json таймстемпов не несут (и полагаться на них не стали бы) → **адаптер таймстемпит момент чтения строки**. `start` = чтение assistant-события с tool_use (в `-p` сообщение приходит целиком ⇒ это ≈ старт исполнения); `duration` = до чтения парного tool_result (матч по `tool_use_id`). Точность телеметрийная, не биллинговая — задокументировать.
- Параллельные tool_use одного сообщения: общий `start`, перекрывающиеся интервалы, сумма `totalDuration` может превысить `wallTime` — свойство измерения, документируем, не боремся.
- Orphan tool_use (процесс умер до result): `duration` = start → выход процесса («работал, пока всё не кончилось»), не фиктивный ZERO.
- Вложенные вызовы сабагентов: в trace — **только верхний уровень** (фильтр по `parent_tool_use_id`), иначе двойной счёт в агрегатах (duration Task-вызова уже включает вложенные). Полная стенограмма — забота сырого потока/лога. Наличие/имя поля проверить на reference-файлах (тема 5); фолбэк для старых CLI без поля — писать всё, что видим.
- `seq` = порядок чтения tool_use-блоков; `ExecutorUsage.tools` сворачивается из trace (один источник истины).

**Live-прогресс — РЕШЕНО: listener-seam в адаптере + два подписчика.**
- Парс-луп адаптера эмитит прогресс как данные: sealed-тип вида `RoundStarted(model, sessionId)` / `ToolStarted(name)` / `RoundFinished(...)`. Это **SPI адаптера**, не доменный порт — порт `StageExecutor` и домен не трогаются. Контракт listener'а дословно списывается с `EngineEventListener`: синхронная доставка, «вернись быстро», исключения глотаются (observability, never an effect).
- Прецедент уже в кодовой базе: `StatusSnapshotHolder` питается из двух источников — `EngineEvent`ы (`StatusEventListener`) и маркеры промптов `DialogConsole` (`SnapshotActivityTracker`, task 6.3). Прогресс исполнителя — третий питатель того же ряда.
- **Подписчик №1 — SLF4J-рендер** (лог = один из рендеров прогресса): консоль ручного запуска получает живую ленту через Logback; в factory loop те же строки уедут в файл/коллектор. Логируем под `AttemptKey` (UX2): старт раунда (модель, session id), каждый top-level tool_use (имя, без input), финал (subtype + сводка токенов); сырые события — DEBUG.
- **Подписчик №2 — обогащение `Activity.Executing`** аддитивной деталью («текущий инструмент, счётчик вызовов») — В SCOPE: закрывает слепую зону `gnomish status` (сейчас на минуты раунда виден только `Executing(since)`), едет той же волной аменда status v1, что и `tokensByModel`.
- Бонус для темы 5: в контрактных тестах прогресс ассертится через listener, а не скрейпингом логов.

**Закрытие нестыковки с темой 1:** «сырец decision-файла в ToolTrace» невозможен — `ToolCall` не несёт payload. Решение: сырое содержимое decision-файла идёт в лог (WARN при невалидном JSON, с полным содержимым), коррелированный по `AttemptKey`. `ToolTrace` остаётся чистым `(seq, tool, start, duration)`. Формулировка в теме 1 исправлена по месту.

## 4. Judge — РЕШЕНО: внутрь scope (`CliJudgeVoter`)

**Решение:** реальный LLM-судья входит в scope — второй адаптер порта `JudgeVoter` на том же CLI-machinery, что и executor (ProcessBuilder, stream-json парсер, телеметрия). Interactive-судья из change 2 остаётся альтернативным биндом.

**Почему внутрь:**
- Гипотеза change («AI-гном проходит QC-цикл») с человеком-судьёй проверяется наполовину: самое ценное звено — «судья нашёл → findings фидбеком → агент починил → судья принял» — не проверяется вовсе.
- Маргинальная цена мала: всё несущее уже готово — порт «один голос» (`JudgeVoter`, мажоритарный цикл в движке: `JudgeVoting`, early-stop, short-circuit на CannotVerify), контрактная сьюта `JudgeVoterContract`, конфиг (`VerifyCheck.Judge`: criteriaFile, model, settings, votes), семантика CannotVerify = infrastructure failure. Процесс-раннер и stream-json парсер строятся для executor'а в любом случае; судья — упрощённый раунд (без feedback-рендера и decision-протокола).
- Труба фидбека готова: findings проигравших голосов уезжают агенту через `Request.feedback` — уже в движке, покрыто тестами.

**Отклонено:**
- *A: человек остаётся судьёй* — QC-петля не автономна (каждый голос ждёт оператора), фидбек-генератор — человек; проверка гипотезы вырождается.
- *C: отложить до api-executor* — судья по природе агентен (должен читать рабочую копию: файлы, grep); честный API-судья = самодельный агентский цикл поверх API, т.е. то, что CLI даёт бесплатно. Native structured output этого не перевешивает.

**Механика `vote()` (рамочно):**
- Промпт = содержимое criteriaFile + TaskContext (title/body/decisions — порт прокидывает его именно за этим, FR7) + инструкция структурного вердикта `{passed, findings[]}`.
- `claude -p`, cwd = workspace root, модель `check.model()`, **read-only allowlist инструментов** (Read/Grep/Glob…) — судья не смеет менять оцениваемую копию; детали → тема 6 (settings).
- Токены голоса из stream-json → `Vote.tokens`; мажоритарность — забота движка, адаптер отвечает ровно на один голос.
- **Инвертированный дефолт деградации:** «вердикт не получен» (непарсабелен / процесс упал) = `CannotVerify`, никогда молчаливый Pass — у executor-протокола позади QC-сетка, судья сам и есть сетка. Дальше готовое: short-circuit чека → infrastructure failure → попытка не горит → эскалация «cannot verify».

**Съём вердикта — РЕШЕНО: финальное сообщение, не файл.**
- Промпт требует последним сообщением выдать только JSON `{passed, findings[]}`; адаптер толерантно извлекает его из result-события stream-json (снять fences, взять первый JSON-объект). Не извлеклось → `CannotVerify`, сырой текст — в лог (WARN, полное содержимое, под `AttemptKey` — та же дисциплина, что у decision-файла).
- Почему не файл: (1) конфликт с read-only судьёй — файл требует точечной Write-дырки в allowlist'е (permission-риск темы 1, умноженный на противоречие духу read-only; у executor'а конфликта не было — он и так пишет в workspace); (2) канал финального сообщения у судьи свободен — его резюме и вердикт одно и то же, аргумент отказа из темы 1 не переносится; (3) result-событие и так парсится и является существенным (тема 3) — не нужны ни temp-каталог, ни env-переменная, ни жизненный цикл файла; (4) симметрия с decision-файлом ложная — дефолты протоколов противоположны (нет сигнала = Completed vs нет вердикта = CannotVerify).
- Деградация безопасна по построению: непарсабельный вердикт не сжигает попытку — CannotVerify → infrastructure failure → эскалация «cannot verify»; цена промаха — внимание человека, не потерянная работа.
- Оговорка та же, что у decision-протокола: пересмотреть (в сторону файла со скоуп-Write), если практика покажет плохую JSON-дисциплину финального сообщения.

**Модель голоса — РЕШЕНО: зеркалируем решение executor'а из темы 3, без клонов логики.**
- `Vote.tokens: @Nullable TokenUsage` → `Vote.tokensByModel: Map<String, TokenUsage>`; пустая map = не зарепорчено (nullability уходит, как у `ExecutorUsage`). `JudgeUsage.perVote` — список таких map'ов: грань «per vote» сохраняется (NFR-C1 — сырое зерно для будущих бюджетов). Обёртка в мелкий рекорд `VoteUsage(tokensByModel)` для читаемости — вкус на этапе propose.
- Ключи — резолвленные model id из `modelUsage` result-события, с тем же фолбэком для старых CLI, что у executor'а. `check.model()` в телеметрию не идёт: это алиас из манифеста, мутирует с его правкой.
- Обоснование: голос судьи под капотом — полноценный агентский раунд, может быть мультимодельным; парсер stream-json, merge и display/total-хелперы — ровно те же, что для executor'а. Один парсер, один merge, никакой второй формы той же телеметрии.
- Interactive-судья репортит пустую map вместо `null`; контрактная форма `PASS_WITHOUT_TOKENS` в `JudgeVoterContract` меняется на «пустая map» — сьюта и так трогается волной 4-польного `TokenUsage`.
- Status v1: `JudgeUsageDto` амендится той же волной, что `UsageDto` (`tokensByModel`) — новых волн не появляется.
- Отклонено: одиночный `TokenUsage` + поле `model` — дешевле на вид, но ломается на мультимодельном раунде и заводит вторую форму той же телеметрии рядом с map'ой executor'а.

**Wiring — РЕШЕНО: манифест-driven дефолт + `--interactive[=executor|judge]`.**
- Рамка: «interactive» — не механизм пайплайна (в `ExecutorType` его нет, и правильно): манифест (§5 Mechanism: `executor.type` + pinned model + settings; judge-чек аналогично пиннит модель) — источник истины о механизме, интерактивные адаптеры change 2 — подмена механизма человеком для supervised-прогона. Runner механизм не выбирает — только явно просит подмену.
- Дефолт (без флагов): стадия `agent-cli` → `CliStageExecutor`, judge-чек → `CliJudgeVoter` (модель/settings из манифеста). Стадия `api` → fail fast на старте («api executor not supported yet»), до любого диалога — в существующую валидационную цепочку (args → pipeline load, exit 3). Рантайм-диспатч/композитный executor не нужны: `api` отсечён на старте, бинд один; композит появится с add-api-executor.
- `--interactive` — явный override, возвращает поведение change 2 целиком (человек играет executor'а и судью). Скоупированные значения `--interactive=executor` / `--interactive=judge` — смешанные прогоны, оба со сценарием: реальный агент + человек-судья (калибровка вердиктов), человек-executor + реальный судья (отладка judge-промпта/парса без оплаты агентских раундов). Парс `--key=value` с валидацией значений уже есть (task 7.1 change 2).
- External-чек всегда интерактивный — реального адаптера в этом change нет, флаг на него не распространяется.
- Платный дефолт осознан: `gnomish run` без флагов на agent-cli манифесте запускает платные CLI-раунды — в этом смысл инструмента, оператор рядом. Предохранитель-подтверждение перед первым платным раундом — НЕ нужен (решено). Альтернатива «дефолт interactive + `--live`» отклонена: обесценивает манифест как источник истины, лишний флаг для «настоящего» режима навсегда.

**Хвост:** `.claude/rules/stage-description.md` §6 говорит «judge — LLM-as-judge via the executor port» — код ушёл к отдельному порту `JudgeVoter` (D2 add-stage-engine); правило поправить.

## 5. Тестовая стратегия — РЕШЕНО: три слоя

**Рамка:** контракт адаптера — с CLI, а не с бэкендом: stream-json генерирует сам CLI независимо от модели за ним. Опора готова: контрактные сьюты всех портов (CLI-адаптеры обязаны их пройти), testing.md (Testcontainers только для E2E), PIT 100% на Java production-код.

**Слой 1 — reference-дампы для парсера.** Записанные дампы реальных stream-json прогонов (`*.reference.json` — терминология проекта, не «golden») как фикстуры юнит-тестов парсера: обычный раунд; раунд с сабагентами (проверить `parent_tool_use_id`); judge-вердикт в финальном сообщении; result с `modelUsage` и без (старый CLI). Закрывает риск темы 3 «формат полудокументирован и дрейфует»: парсер тестируется на байтово-реальных данных. Дампы пишутся руками при работе над change (с чисткой чувствительного); источник — слой 3b.

**Слой 2 — фейковый агент.** Исполняемый скрипт, подменяющий бинарник `claude`: читает аргументы, печатает заданный stream-json, пишет файлы в workspace, выходит с нужным кодом. Требование: путь к бинарнику конфигурируем (заказ в тему 6). Даёт интеграционную честность (реальный ProcessBuilder, пайпы, exit-коды) бесплатно и детерминированно. На нём: контрактные сьюты `StageExecutorContract`/`JudgeVoterContract` для CLI-адаптеров, decision-протокол (файл есть/нет/мусор), verdict-парс (чистый JSON/fences/мусор), убитый процесс, live-прогресс через listener (тема 3). Ollama слой 2 НЕ заменяет: фейк детерминирован и быстр — контрактные сьюты гоняются на нём.

**Слой 3a — local-LLM E2E (реальный CLI + Ollama).** Основной E2E адаптера: с Ollama v0.14+ (янв 2026, нативный Anthropic Messages API) реальный `claude` CLI указывается на локальную модель через `ANTHROPIC_BASE_URL=http://localhost:11434` + `ANTHROPIC_AUTH_TOKEN=ollama` + `ANTHROPIC_DEFAULT_*_MODEL`. Реальный CLI даёт с Ollama то же, что с Anthropic: эмиссию stream-json, permission-механику, exit-коды, агентский tool-цикл — бесплатно и локально. Сценарий: «агент прошёл стадию → судья вынес вердикт»; сценарии тривиальные (создать файл с содержимым X; критерий судьи «файл существует») — чтобы провал слабой локальной модели не читался как баг адаптера. Гейт: наличие живого Ollama, нет — скип с внятным сообщением. Ollama — нативный dev-пререквизит, НЕ Testcontainers: на macOS докеризованный Ollama не получает Metal (CPU-only, непригодно медленно). Заказ в тему 6: адаптер должен уметь пробрасывать env в процесс CLI.

**Слой 3b — платный smoke: ТОЛЬКО ручной запуск.** Отдельный Gradle-таск вне `check`; в CI не ходит никогда; при отсутствии живого залогиненного `claude` падает сразу с внятным сообщением. Главная роль — записать reference-дампы реального бэкенда для слоя 1 (резолвленные model id, cache-токены — Ollama их не даёт: локальные имена моделей, нулевые cache-поля) и подтвердить отсутствие расхождений. Когда запускать (по событиям, не по расписанию):
- обновилась версия `claude` CLI, используемая проектом — формат stream-json дрейфует между версиями;
- парсер меняется под новое/иное поле формата (нужен свежий дамп реального бэкенда, а не гипотеза о нём);
- слой 3a или слой 1 падает на необъяснимом расхождении формата — проверить, реальность это или локальная особенность Ollama;
- перед архивированием change — финальное подтверждение, что адаптер работает против реального бэкенда.

**Чего Ollama-слой не закрывает (и почему 3b остаётся):** форма реального `modelUsage`/cache-токенов (решения темы 3 построены на ней), комплаентность протоколам у сильных моделей (провал локальной 7–30B модели может быть флейком интеллекта, не багом), auth-путь реальной подписки.

## 6. Схема settings — РЕШЕНО: два дома + зашитая политика

**Сортировочный критерий:** «переедет ли настройка вместе с репо на другую машину, не соврав?» Манифест `.gnomish/` — переносимый контракт стадии (стейтлесс-инстансы исполняют стадию идентично на любой машине); всё машинное в нём — ложь.

**Дом 1 — settings-map манифеста** (`executor.settings` стадии, `settings` judge-чека) — только поведение стадии:
- `allowedTools` / `disallowedTools` — списки строк, прозрачно в флаги CLI;
- `maxTurns` — внутрираундовый лимит агентского цикла (родственник `attemptLimit` уровнем ниже; `AutonomyLimits` несёт только attemptLimit — дублирования нет);
- `roundTimeout` — wall-clock лимит процесса раунда (зависший CLI не должен вешать движок); истечение = infrastructure failure (вердикта нет).
- Модель в settings НЕ дублируется — first-class поле (`executor.model`, `check.model()`) → `--model`.

**Дом 2 — конфигурация инсталляции** (`FactoryProperties`/`application.yaml`/env, механизм с change 1):
- путь к бинарнику CLI (дефолт `claude` из PATH) — заказ слоя 2 темы 5 (фейк подменяет бинарник);
- env-проброс в процесс CLI (`ANTHROPIC_BASE_URL` и т.п.) — заказ слоя 3a темы 5 (Ollama);
- temp-каталоги раундов — настройки не требуют (тема 1: адаптер, `Files.createTempDirectory`).

**Не настройки — зашитая политика адаптера:**
- судья строго read-only (Read/Grep/Glob), non-negotiable; `allowedTools` judge-чека может только сужать;
- Write-дырка на `$GNOMISH_DECISION_FILE` для executor-раунда — адаптер сам добавляет точечный allowlist на путь, который сам сгенерировал (закрывает permission-риск темы 1);
- транспортные флаги (`-p`, `--output-format stream-json --verbose`) — внутренность протокола.

**Неизвестные ключи в settings — строгая валидация на старте, силами адаптера.** Загрузчик остаётся опаковым (D5a). Неизвестный ключ = ошибка старта до любого диалога (fail fast в духе загрузки пайплайна): опечатка вроде `allowedTols` иначе тихо меняет поведение стадии. Цена «манифест с ключами будущего api-executor'а не загрузится» мнимая: settings принадлежат конкретному `executor.type`, а `api`-стадии и так отсекаются на старте (тема 4, wiring).

## 7. Общий stage briefing builder — РЕШЕНО: общий секционный рендер, разная политика control-файла

**Проблема:** промпт агентского раунда состоит из тех же секций, что briefing интерактивного executor'а (change 2, `StageBriefing`): task goal, input artifacts, фидбек прошлых попыток (разворот findings), decisions, control-файл. Свой сборщик в CLI-адаптере = клон логики, который разъедется при первой правке. Рендер фидбека в промпт, адресованный сюда темой 2, — это ровно существующий `renderFeedback`.

**Решение из трёх пунктов:**
1. **Секционный рендер извлекается в общий компонент** в отдельном пакете (например `adapter.briefing`) с явным публичным API — module boundaries запрещают CLI-адаптеру лезть во внутренности соседнего `adapter.console`. Секции принимают уже прочитанные данные; чтение файлов из общего рендера убирается. Поведение change 2 не меняется ни на символ. Бонус: `StageBriefing` (162 строки) и так выше файлового лимита 100–120 — разбиение назрело.
2. **Чтение control-файла — у каждого адаптера, реакция на провал разная.** Человеку — заглушка-плейсхолдер (он видит её глазами и сам решает, работать ли); агенту — стоп: нечитаемый control-файл = infrastructure failure ДО старта процесса (попытка не горит, эскалация «cannot execute»). Молчаливый промпт с заглушкой = стадия тихо выполняется без секции Control — verify может пройти, и никто не заметит.
3. **Судья использует подмножество секций:** task goal + decisions (FR7 — решения человека обязаны дойти до судьи) + criteria-файл + инструкция JSON-вердикта. Фидбек прошлых попыток судье НЕ рендерится: голос оценивает текущее состояние против критериев; знание «прошлый раз завалилось» смещало бы вердикт.

CLI-executor поверх общих секций добавляет свой протокольный эпилог — инструкцию decision-файла (тема 1).

## 8. QC-петля глазами агента — РЕШЕНО: три дополнения к промпту и телеметрии

**Петля после решений 1–7 (для контекста):** промпт (секции темы 7 + decision-инструкция) → `claude -p` в workspace → нет decision-файла = Completed (тема 1) → verify (builtin/command — реальные раннеры change 2, judge — CLI-судья темы 4, external — человек) → quality failure → findings в `Request.feedback` → свежий запуск в той же рабочей копии (тема 2) → pass → advancement; исчерпание попыток → эскалация с историей findings.

**Дыра а — rework-преамбула.** Секция фидбека говорит «что упало», но не говорит «рабочая копия уже содержит результат прошлой попытки — дорабатывай, не начинай с нуля». Свежезапущенный агент без памяти (тема 2) может снести наработанное. Решение: промпт ретрая получает rework-преамбулу — часть briefing-эпилога CLI-executor'а (тема 7).

**Дыра б — verify-план в промпте, целиком, включая критерии судьи.** Мотивы: `command`-чеки агент может прогнать сам до завершения раунда — самопроверка экономит попытки (самый дорогой ресурс петли); критерии приёмки — часть ТЗ, а не экзаменационный секрет (по stage-description они обязаны быть конкретными и проверяемыми — их выполнение и есть цель). Контраргумент Гудхарта отвергнут: если критерий можно формально удовлетворить, не решив задачу, — это дефект критерия, и ручные прогоны должны его вскрывать.

**Дыра в — судьба финального сообщения executor-раунда.** Тема 1 сберегла канал под человекочитаемое резюме, но потребителя не назначила. В домен не тащим (резюме — не данные для решений движка; порт стабилен). Решение: текст резюме едет в `RoundFinished` listener-событие (SPI темы 3); SLF4J-подписчик логирует его под `AttemptKey`, оно же видно в live-ленте. Для этого change хранение = лог.

**⚠ Доработка при появлении git-workflow (зафиксировать в proposal будущего change):** резюме раунда сейчас живёт только в логе и live-ленте — когда появится git-workflow (ветка задачи, коммит на попытку), резюме раунда должно класться в коммит-сообщение попытки, чтобы история попыток была читаема из ветки без доступа к логам конкретного инстанса. Это забота git-workflow change, но требование рождено здесь.
