# Решения explore-сессии: add-tracker-port

> Временный файл — решения, принятые в explore-сессии по change `add-tracker-port`
> (вариант A из explore-notes-future-changes.md: сначала tracker-порт в однозадачном
> режиме, factory loop — отдельным change). Не для коммита. Сырьё для proposal/design.

## Контекст

- Однозадачный режим: `gnomish run` берёт задачу из issue трекера вместо ад-хок текста;
  эскалация → Blocked + отчёт-комментарий; resume подбирает решение из комментария.
- Claim-протокол, heartbeat, параллелизм, poll-цикл — НЕ здесь (factory loop).
- Кандидат на попутную нагрузку: реальный external-чек (CI на ветке задачи) через тот же
  GitHub-адаптер + дельта pipeline-config про external-таймауты.

## Решения

### Р1. Абстракция раньше адаптеров (рамка change)

- Порт проектируется **от потребностей фабрики** (что она читает / пишет / меняет в
  трекере), а не от возможностей конкретного трекера. Трекеры очень разные — абстракция
  говорит на языке фабрики, маппинг на реальность трекера — забота адаптера.
- Реализации: **эталонная in-memory** (референс + база контрактных тестов), затем
  GitHub, позже Jira.
- **Инструкция для сторонних разработчиков адаптеров** (пример: Redmine) — часть
  change: абстракция + эталонная реализация + порт-контрактная спека, которую обязан
  пройти любой адаптер (правило testing.md уже требует contract tests для каждого порта).

### Р2. Режимы работы фабрики (уточнение рамки, задано оператором)

- **Автономный режим**: фабрика запущена без указания задач → идёт в трекер → находит
  список задач, готовых к взятию в работу (критерий готовности — статус/тег/иное:
  маппинг адаптера) → берёт первые по приоритету, сколько позволяет лимит параллелизма
  (настройка, напр. 3) → по завершении одной задачи идёт за следующей → пустая очередь:
  ждёт интервал и опрашивает снова (webhook'ов нет — правило «no inbound HTTP»).
- **Явный режим**: фабрике при запуске дают конкретную задачу (id) или несколько (список
  id). Лимит параллелизма действует и здесь: дали 10 задач при лимите 3 → в работе
  первые 3, следующая берётся, когда одна из начатых полностью выполнена.
- Принцип упорядочения очереди («самая приоритетная / первая по id / ещё как-нибудь») —
  не семантика фабрики: адаптер отдаёт готовые задачи уже в порядке очереди, фабрика
  берёт с головы.

### Р3. Политика слота при паркинге — явная настройка фабрики

- Настройка обязана быть прописана в конфиге явно (без скрытого дефолта в коде);
  рекомендованное значение в шаблоне конфига — `hold`.
- **`hold` (дефолт шаблона)**: слот освобождается только при Completed. При
  Escalated/Paused инстанс удерживает claim и сам поллит трекер — не вернул ли человек
  задачу в работу, чтобы продолжить её в том же слоте (тот же worktree, тот же инстанс).
- **`release`**: Escalated/Paused снимают claim; задача припаркована без держателя, слот
  сразу берёт следующую из фида; возвращённую человеком задачу подберёт любой инстанс
  как обычную Ready (resume из ветки задачи — механика git-workflow).
- **Aborted освобождает слот всегда** — запуск мёртв, удерживать нечего; настройка не
  влияет.
- Осознанное следствие `hold`: все N слотов могут повиснуть в ожидании людей — фабрика
  перестаёт брать новые задачи, пропускная способность в руках оператора.

### Р4. Словарь состояний задачи и матрица переходов (принято)

Три уровня, не смешивать:
- **состояние задачи в трекере** — язык координации (виден всем инстансам и людям);
- **исход запуска** (`TaskOutcome` движка: Completed / Paused / Escalated / Aborted) —
  событие, порождающее переход состояния;
- **слот планировщика** (free / busy) — внутреннее дело инстанса, в трекер не пишется.

Словарь состояний задачи:

| Состояние       | Смысл                                                                                                                          | Атрибуты                                                                                           |
|-----------------|--------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------|
| `Ready`         | ничья; критерий готовности выполнен; доступна для claim (новая или возвращённая — различаются наличием ветки `gnomish/<task>`) | —                                                                                                  |
| `Working`       | в работе у конкретного инстанса (claim)                                                                                        | инстанс-держатель                                                                                  |
| `AwaitingHuman` | ждёт человека                                                                                                                  | причина: `escalation(report)` \| `checkpoint(passedStage)`; держатель: `held(инстанс)` \| `parked` |
| `Finished`      | работа сдана (Completed); фабрика задачу больше не трогает                                                                     | —                                                                                                  |

Вне словаря: задачи, не проходящие критерий готовности, фабрика не видит; закрытие
issue после merge — жизнь трекера, не фабрики. `Paused` — НЕ отдельное трекерное
состояние: это исход `Paused(passedStage)` от стадии с `advancement: manual`
(checkpoint из stage-description), в трекере он выглядит как `AwaitingHuman(checkpoint)`.

Матрица переходов:

| # | Из                    | Событие                                           | В                                         | Инициатор                                                        | Настройка/условие                                         |
|---|-----------------------|---------------------------------------------------|-------------------------------------------|------------------------------------------------------------------|-----------------------------------------------------------|
| 1 | Ready                 | инстанс взял задачу (автономный фид или явный id) | Working                                   | фабрика                                                          | claim                                                     |
| 2 | Working               | исход Completed                                   | Finished                                  | фабрика                                                          | всегда; слот свободен                                     |
| 3 | Working               | исход Escalated(report)                           | AwaitingHuman(escalation, held)           | фабрика                                                          | слот-политика = hold                                      |
| 4 | Working               | исход Escalated(report)                           | AwaitingHuman(escalation, parked)         | фабрика                                                          | слот-политика = release                                   |
| 5 | Working               | исход Paused(passedStage)                         | AwaitingHuman(checkpoint, held \| parked) | фабрика                                                          | та же слот-политика                                       |
| 6 | Working               | исход Aborted                                     | Ready                                     | фабрика (best-effort: трекер обычно жив, даже когда git отказал) | всегда; слот свободен; комментарий с причиной             |
| 7 | AwaitingHuman(held)   | человек ответил и вернул в работу                 | Working — тот же инстанс, тот же слот     | человек; держатель замечает поллингом                            | hold                                                      |
| 8 | AwaitingHuman(parked) | человек ответил и вернул в работу                 | Ready                                     | человек                                                          | release; далее переход 1 любым инстансом, resume по ветке |
| 9 | AwaitingHuman         | человек закрыл задачу (отказался продолжать)      | вне мира фабрики                          | человек                                                          | held-держатель замечает и освобождает слот                |

Открытый острый случай (не в матрице): человек закрыл/отозвал задачу в состоянии
Working, под работающим гномом — когда фабрика обязана это заметить (граница раунда?)
и что делает с работой. Обсудить отдельно.

### Р5. Политика Aborted (принято)

- **Что такое Aborted**: инфраструктурная смерть запуска — движок не смог durable
  записать состояние (strict persist) либо упал сам. Задача «не виновата»: это не
  качество, попытки стадии не тратятся.
- **Протокол одного аборта** («вежливая смерть», развитие заметок future-changes):
  ERROR в лог → best-effort **структурный** комментарий в трекер (машинно-узнаваемый
  маркер: причина, инстанс, время) → снять claim → задача в Ready. Слот освобождается
  всегда (Р3).
- **Backoff на повторный подбор**: счётчик абортов и время последнего аборта
  восстанавливаются из структурных комментариев фабрики в трекере — трекер, потому что
  отказавшим сервисом мог быть сам git, а комментарии переживают инстансы (сценарий
  «другая фабрика взяла и снова аборт» покрыт: счётчик общий для всех инстансов).
  Разделение ролей: **адаптер отдаёт факты** (abortCount, lastAbortAt при выдаче
  задачи), **политику применяет ядро** — задача в Ready с неистёкшим backoff невидима
  автономному фиду. Это фильтр фида, НЕ отдельное трекерное состояние. Форма backoff
  (например, экспоненциальный с потолком) и параметры — конфиг фабрики.
- **Предохранитель**: K абортов подряд без прогресса → вместо возврата в Ready задача
  уходит в AwaitingHuman с инфраструктурным отчётом («cannot persist: причина, история
  абортов всех инстансов»). K — конфиг. «Без прогресса» = счётчик сбрасывается первым
  durable-закоммиченным раундом после claim (аборты, перемежающиеся реальным
  прогрессом, предохранитель не взводят).
- **Явный режим — мандат оператора**: явное указание id игнорирует backoff и берёт
  даже задачу, припаркованную предохранителем (оператор починил окружение и вручную
  перезапускает — это и есть resume инфраструктурной эскалации).
- **Экономика**: аборт может случиться ПОСЛЕ работы агента (раунд выполнен, persist
  упал) — каждый ретрай перевыполняет раунд и жжёт токены; backoff и K защищают бюджет,
  а не только от шума в трекере.
- **Двойной отказ** (git и трекер мертвы): комментарий не ушёл, аборт виден только в
  логах, claim повисает → снимается протоколом stale-claim/heartbeat (factory loop,
  обсуждается там).
- **Дельта к матрице Р4**: переход 6 расщепляется:
  6a `Working → Ready` (Aborted, счётчик+1 < K; + структурный комментарий);
  6b `Working → AwaitingHuman(escalation, infra)` (Aborted, счётчик+1 ≥ K).

### Р6. Отзыв задачи под работающим гномом (принято)

- **Обобщение**: закрытие задачи человеком — частный случай «отзыва». На границе
  каждого раунда инстанс делает одну проверку «задача всё ещё наша и жива»: не закрыта
  ли, не снят/не перехвачен ли claim, не переведена ли человеком в другое состояние.
  Один вопрос к трекеру — все случаи отзыва разом.
- **Точка обнаружения v1 — граница раунда** (точка покоя: state закоммичен, дёшево и
  детерминированно). Отмена посреди раунда (kill гнома) — отложена: требует
  cancel-шва в agent-cli executor; кандидат на будущее, когда длинные дорогие раунды
  станут болью. Латентность обнаружения = длительность одного раунда — осознанная цена.
- **Действия при «задача закрыта»**: salvage-коммит незакоммиченной работы (дёшево,
  ничего не теряем) → push best-effort → структурный комментарий «работа остановлена:
  задача закрыта» → снять claim → слот свободен. Трекерное состояние не меняем —
  закрытая задача вне мира фабрики. Ветка `gnomish/<task>` остаётся (аудит), worktree —
  как у Escalated: сохраняется; политики очистки по возрасту — territory factory loop.
- **Исход запуска** — не Aborted (инфраструктура жива) и не Escalated: отзыв — это
  третий, штатный внешний исход остановки. Как его моделировать (новый вариант
  TaskOutcome? исход раннера поверх движка?) — вопрос design, не explore; помечено.

### Р7. Явный режим: диспозиция по состояниям задачи (принято)

- **Принцип мандата**: явное указание id — распоряжение оператора. Критерий готовности
  (тег/статус) — фильтр только для автономного фида; явный режим берёт задачу, даже
  если тег готовности не стоит. Backoff и предохранитель Aborted мандат тоже пробивает
  (Р5), давая **одну** попытку в обход: счётчик абортов не сбрасывается мандатом —
  сброс только по общему правилу «первый durable-раунд после claim» (прогресс).
Диспозиция по состояниям:

| Состояние задачи                              | Действие явного режима                                                                              |
|-----------------------------------------------|-----------------------------------------------------------------------------------------------------|
| `Ready` (и «не Ready» по критерию готовности) | claim → работа (мандат)                                                                             |
| `AwaitingHuman` (escalation)                  | запрос на resume: собрать решение (порядок ниже) → claim → продолжение из ветки                     |
| `AwaitingHuman` (checkpoint)                  | то же, но нужно подтверждение «продолжай», не решение                                               |
| `AwaitingHuman` (infra, предохранитель Р5)    | resume инфра-эскалации: одна попытка в обход предохранителя                                         |
| `Working` (claim другого инстанса)            | отказ с внятной ошибкой, называющей держателя; кража claim — territory stale-claim/heartbeat (loop) |
| `Finished`                                    | «уже сделана» — скип с отчётом                                                                      |
| закрыта / не существует                       | скип с внятной ошибкой                                                                              |

- **Порядок сбора решения при resume** `AwaitingHuman`:
  1. непотреблённый ответ человека в трекере (комментарий-решение после эскалационного
     отчёта — спаривание эвристикой адаптера) → используем;
  2. нет ответа в трекере, но есть TTY → интерактивный диалог (как manual-run), и
     введённое решение **зеркалируется комментарием в трекер** — след решения всегда
     остаётся в трекере, независимо от способа ввода;
  3. нет ответа и нет TTY (headless) → задачу не трогаем, остаётся AwaitingHuman,
     внятный отчёт «ответь в трекере и перезапусти».
- **Запасной выход при зависшем Working**: до появления heartbeat (loop) задача
  умершего инстанса вечно висит в Working; ручной выход — человек сам переводит
  состояние в трекере (человеческие переходы легитимны), после чего явный режим её
  возьмёт. Задокументировать в инструкции оператора.
- **Несколько id**: невзятые задачи (чужой Working, Finished, закрытые) скипаются с
  отчётом, остальные обрабатываются; итоговая сводка по всем перечисленным задачам.
  Агрегация exit-кодов мультизадачного запуска — вопрос design, не explore.

### Р8. Ожидание решения — гонка «консоль vs трекер» (принято)

- Интерактивный диалог ожидания решения (Р7, шаг 2) слушает **два источника
  одновременно**: ввод с TTY и поллинг трекера (интервал — конфиг). Кто первым дал
  непустой ответ — тот и победил; второй источник отменяется. Диалог явно говорит
  оператору: «ответь здесь или комментарием в трекере — слежу за обоими».
  (Механика — два виртуальных потока, first-wins; деталь design.)
- **Единый механизм подтверждения потребления**: потребив решение из ЛЮБОГО источника,
  фабрика постит ack-комментарий «acting on decision: <текст>». Для консольного ввода
  это и есть зеркалирование из Р7; для трекерного ответа — подтверждение «услышала,
  продолжаю с этим». Зеркало и ack — одна и та же операция.
- Ack-комментарий решает гонку двух разных ответов (в консоли одно, в трекере другое):
  фабрика действует по первому увиденному, ack фиксирует в трекере, ПО КАКОМУ решению
  она работает — аудит однозначен, второй ответ остаётся обычной репликой в обсуждении.
- Ack — якорь для эвристики спаривания: при будущих resume адаптер ищет
  решения-комментарии ПОСЛЕ последнего ack'а; непотреблённые устаревшие реплики до него
  никогда не применятся к новой эскалации.
- Оператор вышел из диалога, не ответив (Ctrl-D), а потом ответил в трекере → штатный
  путь: задача осталась AwaitingHuman, ответ подберёт следующий запуск (авто или явный)
  по шагу 1 порядка Р7.

### Р9. Семантика Finished (принято)

- Логический `Finished` = «работа сдана на проверку»: движок вернул Completed, ветка с
  результатом запушена (позже — PR). Это НЕ «задача закрыта»: закрытие происходит после
  merge и принадлежит жизни трекера/людей, не фабрике.
- Физика отображения — маппинг адаптера (конфиг): GitHub — например, label
  `gnomish:review`, issue остаётся открытым; Jira — transition в «In Review»;
  in-memory — поле состояния.
- При переходе в Finished фабрика пишет финальный отчёт-комментарий (сводка: стадии,
  попытки, ссылка на ветку/PR, usage — состав в design) и после этого **задачу больше
  никогда не трогает**. Повторный прогон сделанной задачи = новая задача, не resume (Р7).

### Р10. Снимок задачи при взятии в работу (принято)

- При ПЕРВОМ claim фабрика читает id/title/body один раз → снимок уезжает в
  `TaskContext` и `task.json` (git-workflow FR3). Дальше правда о задаче — снимок.
- Редактирование issue под живой/припаркованной задачей на неё НЕ влияет. Хочешь
  повлиять на выполняющуюся задачу — комментарий-решение (при эскалации) или закрыть
  задачу (отзыв, Р6) и завести новую.
- При resume снимок НЕ перечитывается — собираются только решения. Свойства: контекст
  гнома не меняется молча посреди задачи; resume полностью восстановим из ветки
  (правило stage-description); title в status-выводах — из снимка.
- **Обязательно прописать в пользовательской документации проекта**: правки issue под
  взятой в работу задачей на неё не влияют; канал влияния — решения (или отзыв задачи
  и новая постановка). Пользователь должен знать это поведение заранее.

### Р11. Трекер — монополия фабрики (принято)

- Прямой доступ к трекеру есть только у ядра фабрики (через порт/адаптер). Гном
  (executor) НИКОГДА: credentials трекера не попадают в окружение процесса гнома,
  ничего трекерного — в промпты. Зеркало push-монополии git-workflow NFR-S1.
- Инструментарий гнома для работы с трекером — опосредованный: контекст задачи (снимок
  + решения) приходит в промпте; вопрос человеку — через `DecisionNeeded`; отчёты и
  статусы в трекер несёт фабрика.
- Статусы задач меняют только фабрика и человек (инициаторы в матрице Р4) — гном не
  является инициатором ни одного перехода.

### Р12. Порт `Tracker`: скоуп и гранулярность (принято)

- **Один порт** (рабочее имя `Tracker`), слой application — рядом с `TaskRepository`
  (git-workflow D1). Резка на узкие порты (фид / координация / переписка) отклонена:
  адаптер трекера всегда реализует все роли сразу, потребитель один (ядро), а цель Р1
  (одна контрактная спека + одна эталонная реализация + одна инструкция для сторонних
  разработчиков) достигается проще с единым интерфейсом.
Операции v1 (выведены из Р2–Р11; сигнатуры — design):

| Операция                          | Что делает                                                                         | Откуда требование                             |
|-----------------------------------|------------------------------------------------------------------------------------|-----------------------------------------------|
| `listReady(limit)`                | Ready-задачи по критерию готовности, в порядке очереди адаптера, с фактами абортов | Р2 (автономный фид), Р5                       |
| `fetchTask(ref)`                  | снимок (title/body) + логическое состояние + держатель claim + факты абортов       | Р7 (мандат), Р6 (проверка «наша и жива»), Р10 |
| `collectDecisions(task)`          | решения-комментарии после последнего ack; спаривание — эвристика адаптера          | Р7, Р8                                        |
| `claim(task, instanceId)`         | захват; результат: успех \| занята (кем)                                           | Р2, Р4 переход 1                              |
| `release(task)`                   | снять claim без смены состояния задачи                                             | Р3 (release-политика), Р6                     |
| `park(task, reason, report)`      | → AwaitingHuman (escalation \| checkpoint), отчёт-комментарий                      | Р4 переходы 3–5, Р5 предохранитель            |
| `finish(task, summary)`           | → Finished + финальный отчёт                                                       | Р4 переход 2, Р9                              |
| `recordAbort(task, cause, ...)`   | структурный маркер аборта + возврат в Ready (одна операция)                        | Р5, Р4 переход 6a                             |
| `acknowledgeDecision(task, text)` | ack «acting on decision: …»                                                        | Р8                                            |
| `postNote(task, text)`            | прочие сообщения (напр. «остановлено: задача закрыта»)                             | Р6                                            |

- **Контрактные свойства** (ядро порт-контрактной спеки, обязательны любому адаптеру):
  - `listReady` не выдаёт Working/AwaitingHuman/Finished и не прошедшие критерий
    готовности; backoff НЕ фильтрует — это политика ядра (Р5: адаптер — факты);
  - **наблюдаемая атомарность claim**: два конкурентных claim одной задачи → ровно
    один успех; как адаптер этого добивается (verify-after-write и т.п.) — его дело;
  - **round-trip структурных маркеров**: после `recordAbort` факты абортов читаются
    обратно любым инстансом; после `acknowledgeDecision` `collectDecisions` пуст до
    нового ответа человека. Формат маркеров в теле комментария — свобода адаптера при
    соблюдении round-trip;
  - рендер доменных отчётов (EscalationReport → текст) — забота ядра: порт принимает
    готовый текст + структурные поля, адаптер доменную модель движка не знает.
- Эволюция: heartbeat/stale-claim добавят операции в этот же порт в factory-loop
  change; абстракция расширяема, v1 их сознательно не содержит.

### Р13. GitHub-адаптер: физический маппинг (подтверждён исследованием API 2026-07-18; имена label'ов — обсуждаются)

- **Принцип: labels — состояние для людей, структурные комментарии — факты для машин.**
  Логические состояния Р4 отображаются взаимоисключающими label'ами с настраиваемыми
  именами (дефолты): `gnomish:ready`, `gnomish:working`, `gnomish:awaiting-human`,
  `gnomish:review` (= Finished). Адаптер поддерживает взаимоисключаемость (снимает
  старый при переходе). Координационные факты (кто держит claim, аборты, ack'и) — в
  структурных комментариях, не в label'ах.
- **Критерий готовности (дефолт)**: issue открыт + label `gnomish:ready`. Порядок
  очереди (дефолт): по возрастанию номера issue (старые первыми). Оба — конфиг адаптера.
- **Projects v2 отклонён для v1**: требует настроенного проекта, GraphQL и органовых
  прав — тяжёлая предпосылка; labels работают на любом репозитории из коробки.
  Возможная опция адаптера в будущем, не сейчас.
- **Claim без CAS**: у GitHub нет транзакций — атомарность по наблюдению (Р12)
  достигается lease-паттерном: поставить label + написать структурный
  claim-комментарий (инстанс, время) → перечитать claim-комментарии после последней
  границы (release/park/abort) → **победил самый ранний comment id** (серверный
  тотальный порядок GitHub); проигравший аннулирует свой маркер и отступает. Детали и
  граничные случаи — design; контрактный тест гонки обязателен (эталонная in-memory
  реализация умеет симулировать).
- **Человеческие переходы** (вернуть задачу в работу): ответ-комментарий + перестановка
  label `gnomish:awaiting-human` → `gnomish:ready` («перемещение — манипуляция
  состоянием, не команда в тексте»). Просто, видно в UI, попадает в инструкцию
  оператора. Закрытие issue = отзыв (Р6); держатель claim'а на закрытой задаче её
  отпускает.
- Привязка репо/имена label'ов — конфиг; credentials — только окружение фабрики (Р11),
  размещение конфига (`.gnomish/` проекта vs конфиг фабрики) — решить в design.
- **Дельты по итогам исследования API (2026-07-18)**:
  - взаимоисключаемость label'ов — точечными `POST /labels` + `DELETE /labels/{name}`,
    НЕ `PATCH` с полем `labels` (тот заменяет весь набор → lost update при
    конкурентной правке человеком);
  - **assignee не годится для координации**: все инстансы ходят под одним токеном и
    выглядят одним пользователем — держателя claim по assignee не различить. Assignee —
    опциональная витрина для людей, не источник правды (источник — claim-комментарий);
  - фид — через `GET /repos/{o}/{r}/issues?state=open&labels=...&sort=created&
    direction=asc` (List, не Search): Search API — отдельный лимит 30 req/min и
    eventual consistency индекса; дефолт очереди «старые первыми» даёт сам GitHub;
  - issue types (org-only) и Projects v2 (GraphQL + известный баг автоматизации
    single-select) — не привязываться, подтверждено;
  - issue lock не использовать — запретит людям комментировать (убивает канал решений);
  - серверный тотальный порядок для lease-паттерна подтверждён: комментарии по
    умолчанию отдаются по возрастанию id; условных записей (CAS/If-Match) у GitHub
    нет — verify-after-write безальтернативен.
- **Имена label'ов (принято 2026-07-19)**: принцип — логические имена (Р4) говорят на
  языке фабрики, label'ы — на языке человека («что мне с этим делать»). Префикс
  `gnomish:` (согласован с `.gnomish/`). Дефолты: `gnomish:ready`, `gnomish:working`,
  `gnomish:needs-human` (ОДИН label для AwaitingHuman, причина escalation/checkpoint —
  в отчёте-комментарии; разделение на два — возможное будущее), `gnomish:delivered`
  для Finished (`review`/`done` отвергнуты: пайплайн конфигурируем — ревью/мерж могут
  быть автоматизированы внутри фабрики, label называет только инвариант «сдано»; при
  авто-merge пайплайне `Closes #N` в PR закрывает issue силами GitHub — закрытие
  остаётся жизнью трекера по Р9). Все имена — конфиг адаптера: у оператора может
  существовать настройка проекта/label'ов, которую он не хочет или не может менять.
- **Провижининг label'ов (принято; дополнено 2026-07-20)**: адаптер при старте
  идемпотентно создаёт отсутствующие label'ы с цветами и description-подсказкой
  оператору (~100 символов). **Цвета — конфиг с дефолтами**, как имена: каждый label
  в `tracker.labels` — объект `{name, color}` (hex без `#`, валидация при загрузке);
  дефолты: ready — зелёный `2ea44f`, working — синий `1f6feb`, needs-human — красный
  `d73a4a`, delivered — фиолетовый `8250df`. **Цвет применяется ТОЛЬКО при создании**
  label'а; существующие не перекрашиваются (уважение к ручным манипуляциям оператора:
  перекрасил в UI — фабрика не воюет; следствие в инструкцию: смена цвета в конфиге
  существующий label не перекрасит). Description — не конфиг v1 (ручки по боли, Р20).

### Р14. Факты GitHub API — экономика поллинга и лимиты (исследование 2026-07-18)

- **Состояния issue**: `state` = open/closed; `state_reason` = completed / not_planned /
  duplicate / reopened / null. Два реальных состояния — словарь Р4 на них не ложится,
  отображение состояний — label'ами (Р13). `closed` = отзыв (Р6) независимо от
  reason; `state_reason` стоит включать в отчёт отзыва (контекст человеку).
- **Первичный лимит**: 5 000 req/час на PAT (15 000 Enterprise Cloud; GitHub App
  масштабируется от 5 000).
- **ETag-поллинг бесплатен**: условный GET с `If-None-Match` → `304 Not Modified` НЕ
  тратит лимит. Фид (Р2), hold-поллинг (Р3), ожидание решения (Р8) при неизменном
  состоянии почти бесплатны; интервалы поллинга можно выбирать смело. ETag — на
  страницу выдачи, наши опросы однострочные (issue или первая страница фида).
- **Вторичные лимиты на запись**: ≤80 content-creating req/мин и ≤500/час; write = 5
  поинтов при потолке 900 поинтов/мин на endpoint. Переход состояния = 2–3 записи
  (label + комментарий) — штатная работа с запасом; abort-шторм упёрся бы в 500/час —
  ещё один довод за backoff Р5.
- Комментарии: `id` int64, есть `since`, пагинация до 100/страницу — `collectDecisions`
  и чтение фактов абортов дёшевы.
- Новое в GitHub Issues (GA апрель 2025): sub-issues (иерархия), issue types,
  advanced search (AND/OR) — sub-issues как «эпик → подзадачи гномам» — потенциал
  далёкого будущего, не v1.

### Р15. Projects v2 и управление фабрикой через доску (принято 2026-07-19)

- **Сосуществование**: labels и Projects v2 ортогональны (label — на issue, статус — на
  карточке проекта); адаптер проекты не трогает, project-права токену не нужны.
  Витрина «фабрика → доска» дёшева: встроенный auto-add workflow проекта затягивает
  issues по label-фильтру (число правил ограничено планом GitHub).
- **Подводный камень (обязателен в инструкции оператора)**: доска — параллельная
  вселенная; перетаскивание карточки label'ы НЕ меняет, фабрика этого не видит.
  Источник правды — labels, человеческие переходы — перестановка label на issue.
- **Три пути «доска → фабрика», решение**:
  - **Путь А — Projects v2 как источник правды адаптера: отклонён для v1** с
    задокументированными причинами: GraphQL не фильтрует карточки по значению поля на
    сервере (фид = выкачать все карточки), у GraphQL нет условных запросов (ETag/304
    не работает — каждый опрос жжёт лимит), два известных бага (single-select
    визуально не двигает карточку; updateProjectV2ItemFieldValue теряет запись при
    включённых встроенных автоматизациях). Координационные факты всё равно остались бы
    комментариями. Зарезервированная опция будущего: пересмотреть, если GitHub даст
    серверную фильтрацию и починит баги.
  - **Путь Б — мост «доска → labels» на стороне оператора: ПРИНЯТ как рецепт.**
    Триггера Actions на project_v2_item нет (org-webhook = входящий HTTP → инфра
    оператора), поэтому мост = cron-Action (минимальный интервал schedule 5 минут).
    Человеческих переходов мало (Р4): достаточно одностороннего правила «карточка в
    колонке X → поставить `gnomish:ready`» — скрипт на несколько строк с `gh` CLI,
    без двусторонней синхронизации и семантики конфликтов.
    **Reference-workflow делаем сразу в составе change** (артефакт при инструкции
    оператора) — не откладывать, чтобы не забыть.
  - **Путь В — labels правда, доска витрина: v1-дефолт** (как в Р13).
- Латентность моста (cron 5 мин) сопоставима с интервалами поллинга фабрики — приемлемо.

### Р16. Идентичность задачи (принято 2026-07-19)

- **Канонический taskId**: `github:owner/repo#42` (шаблон `<tracker>:<координаты>`;
  будущий Jira — `jira:...`). Одна каноническая строка, три производных:
  - `task.json`, логи (MDC), структурные комментарии — каноника как есть;
  - ветка/worktree — существующий sanitize git-task-persistence:
    `gnomish/github-owner-repo-42` (движок и git-workflow не меняются);
  - человеку (`status`, отчёты) — каноника + title из снимка (Р10).
- **Коллизии закрыты по построению**: между трекерами — префикс; между репозиториями —
  `owner/repo` в id (репо с issues может быть не тем же, что целевой репо проекта);
  с ad-hoc — префикс `manual-`. Charset `--task-id` (manual-run) не пропускает `:`/`#`
  — каноническую форму трекера руками не подделать.
- **Resume «из ветки одной» сохраняется**: каноника в `task.json`; адаптер парсит свой
  же формат обратно в координаты issue (внутренний round-trip адаптера; для ядра id
  opaque — Р12 не нарушен).
- **Полный id, не короткий** (`gh-42` отклонён): id вечен (вморожен в ветки и
  комментарии), привязка — изменяемый конфиг; короткий id коллизировал бы задним
  числом при второй привязке. Краткость — на вводе: в явном режиме CLI оператор
  печатает `42`/`#42`, адаптер разворачивает в канонику по привязке (детали — CLI,
  пункт 4 очереди).
- **Известные ограничения v1** (в инструкцию автора адаптера / оператора, принято):
  - sanitize-коллизия на краю: `github:a/b-c#1` и `github:a-b/c#1` → одна ветка;
    внутри одной привязки невозможно, ловится только при двух привязках с путаемыми
    именами репо в один целевой репозиторий — задокументировать, не чинить;
  - `--task-id github-owner-repo-42` (sanitized-форма) руками ввести можно и получить
    коллизию ветки — резервированных префиксов не городим, строчка в инструкции.
- **`status`**: колонка task = каноника + title; косметика коротких форм — потом.

### Р17. External-чек CI — отдельный change, отложен до работ по QC (принято 2026-07-19)

- Из add-tracker-port выносится полностью. Делать отдельным change'ом, когда дойдём
  до QC и его задач (первая реализация типа `external` из stage-description).
- Рамка будущего change'а (зафиксировано, чтобы не передумывать заново):
  - GitHub Checks API `GET /commits/{sha}/check-runs` (Actions пишет сюда; legacy
    combined status — не v1, по запросу); ETag/304 работает — поллинг дёшев;
  - чек привязан к SHA вершины ветки на момент сабмита, не к «ветке вообще»;
  - неудавшийся push (в git-workflow он best-effort) = infra failure чека — CI нечего
    проверять;
  - критерий прохождения: конфиг называет имена ожидаемых check-run'ов, «все названные
    completed + success»; «ждать всех подряд» отклонено (новый чужой чек подвесит
    стадию, неполный список = ложный pass);
  - дельта pipeline-config: у `external` появляются `pollInterval`, `timeout`,
    `timeoutClass: quality | infra` (дефолт quality — как в stage-description).

### Р18. Границы change'а add-tracker-port (принято 2026-07-19)

- **Принцип резки**: tracker-port = «одна задача из трекера, от claim до сдачи, любым
  инстансом»; factory loop = «много задач и много инстансов одновременно». Всё, что
  существует только ради параллелизма и долгоживущего процесса — в loop.
- **Входит в add-tracker-port**:
  - порт `Tracker` (все операции Р12) + in-memory эталон + порт-контрактная спека;
  - GitHub-адаптер: labels, lease-claim, структурные комментарии, провижининг (Р13);
  - однозадачный явный режим: `gnomish run` с одним ref → claim → работа →
    finish/park/abort; снимок при claim (Р10); отзыв на границе раунда (Р6);
  - resume: сбор решений, ack, гонка «консоль vs трекер» (Р7/Р8);
  - Aborted: recordAbort, факты в комментариях, K-предохранитель (Р5) — предохранитель
    срабатывает в момент аборта (выбор 6a/6b), фид ему не нужен;
  - **однозадачный авто-режим (потребитель `listReady`)**: `gnomish run` БЕЗ ref =
    «возьми голову очереди, сделай одну задачу, завершись». Закрывает traceability
    операции, комплектует инструкцию автора адаптера, даёт фабрику «на минималках»
    до loop (оператор крутит cron'ом) — догфудинг порта немедленно;
  - инструкции: оператору (+мост-workflow Р15) и автору адаптера (Р1).
- **Уходит в factory loop** — вынесено в отдельный файл
  `explore-notes-factory-loop.md` (бэклог, чтобы не потерять).
- Слот-политика Р3 целиком в loop: «слот» — сущность планировщика; в однозадачном
  режиме при эскалации — TTY-диалог Р8 либо headless-завершение с отчётом (Р7).
- Зависимость от add-git-workflow снята: реализован и заархивирован 2026-07-19.

### Р19. CLI-поверхность: подкоманда `gnomish take` (принято 2026-07-19)

- **Трекерный режим — отдельная подкоманда, НЕ флаги `run`**: упаковка другого режима
  работы (поход в трекер, claim, диспозиция Р7) в флаг команды с контрактом «одна
  ad-hoc задача» плодила бы третий `--task-*` флаг и путаницу. Спека manual-run не
  меняется, валидационная матрица `run` остаётся как есть.
- **Формы**:
  - `gnomish take <ref>` — явный режим (Р7): `42` / `#42` / полная каноника Р16;
    короткие формы разворачиваются по привязке; каноника с чужим репо — ошибка;
    старт или resume решается состоянием задачи (Р7), отдельный resume-флаг не нужен;
  - `gnomish take` (голая) — авто-режим Р18: голова очереди `listReady`, одна задача,
    завершение. Явный `--next` не нужен: слово take само объявляет намерение,
    опасности «забытого флага» (как у пустого `run`) нет.
- Имя — глагол домена («взять задачу», лексика Р2/Р4); будущая симметрия: `run` —
  ад-хок, `take` — одна задача из трекера, loop добавит команду конвейера
  (`serve`/`loop` — имя решится там).
- **Словарь флагов `take`**: общие с `run` `--dir`, `--interactive[=...]`, git-флаги
  `--base` (старт), `--discard-work` (resume при diverged). `--mode` ОТСУТСТВУЕТ —
  трекер всегда git mode. Ad-hoc группа (`--task`, `--task-file`, `--task-id`,
  `--resume`) отсутствует. `--from-stage` — вопрос design (для Ready-старта).
  Голая форма несовместима с модификаторами старта конкретной задачи (`--base` —
  design уточнит: какая задача — решает очередь).
- Rejected: `--task-ref`/`--tracker-task`/`--issue`/`--ticket`/`--take` как флаги
  `run` (issue/ticket ещё и ломают язык абстракции — в словаре порта «task»).

Сводные таблицы флагов (итог):

`gnomish run` — ад-хок задача (без изменений, как в спеке manual-run):

| Флаг                             | Группа                    | Несовместим с                                              | Комментарий                                                            |
|----------------------------------|---------------------------|------------------------------------------------------------|------------------------------------------------------------------------|
| `--task <текст>`                 | А: источник (ровно один)  | остальной группой А                                        | новая задача из текста                                                 |
| `--task-file <файл>`             | А: источник               | остальной группой А                                        | новая задача из файла                                                  |
| `--resume <id>`                  | А: источник               | группа А; `--mode in-place`; `--base`, `--task-id`, `--from-stage` | продолжить ad-hoc задачу из ветки                              |
| `--task-id <id>`                 | Б: модификатор старта     | только с `--task`/`--task-file`                            | свой id; `:`/`#` не пропускает — канонику трекера не подделать         |
| `--from-stage <стадия>`          | Б: модификатор старта     | не с `--resume`                                            | начать с названной стадии                                              |
| `--base <ref>`                   | Б: модификатор старта     | git mode, только старт                                     | база для ветки задачи                                                  |
| `--dir <путь>`                   | В: окружение              | —                                                          | целевой проект (default: cwd)                                          |
| `--mode git\|in-place`           | В: окружение              | in-place: `--base`, `--resume`, `--discard-work`           | default `git`                                                          |
| `--discard-work`                 | В: окружение              | только с `--resume`                                        | сброс при разъехавшихся ветках                                         |
| `--interactive[=executor\|judge]`| В: механизмы              | —                                                          | ортогонален всему                                                      |

`gnomish take` — задача из трекера (новое, всегда git mode):

| Форма / флаг                     | Роль                | Несовместим с                    | Комментарий                                                                  |
|----------------------------------|---------------------|----------------------------------|------------------------------------------------------------------------------|
| `take <ref>`                     | явный режим (Р7)    | —                                | `42`/`#42`/каноника; старт или resume — по состоянию; чужой репо = ошибка    |
| `take` (голая)                   | авто-режим (Р18)    | модификаторами старта (`--base`) | голова очереди, одна задача, завершение                                      |
| `--dir <путь>`                   | окружение           | —                                | общий с `run`                                                                |
| `--base <ref>`                   | модификатор старта  | голой формой                     | база ветки при старте новой задачи                                           |
| `--discard-work`                 | resume              | —                                | при разъехавшихся ветках, как у `run --resume`                               |
| `--interactive[=executor\|judge]`| механизмы           | —                                | общий с `run`                                                                |
| `--from-stage`                   | под вопросом        | —                                | design: имеет смысл только для Ready-старта                                  |

`--task-id`, `--resume`). В `run` — ни одного нового флага.

### Р20. Словарь конфигурации v1 (принято 2026-07-20)

- **Принцип размещения**: что должно совпадать у всех инстансов одного проекта — в
  `.gnomish/` проекта; темп и личность инстанса — конфиг фабрики; секреты — только
  окружение (Р11). Критерий допуска в `.gnomish/`: **только то, что безопасно
  унаследовать при форке** (имена label'ов — да, K — да, привязка — почти, см. форк;
  идентичность — никогда).
- **`.gnomish/config.yaml`, новая секция `tracker`**: `type` (нет секции = `take`
  недоступна, `run` живёт как жил), `repo` (репо issues, может отличаться от целевого),
  `labels.{ready,working,needs-human,delivered}` (дефолты `gnomish:*`),
  `abortThreshold` (K из Р5, дефолт 3 — общий для всех инстансов, т.к. предохранитель
  считается по общим комментариям).
- **Конфиг фабрики**: `gnomish.instance-name` (дефолт hostname),
  `gnomish.tracker.decision-poll-interval` (дефолт ~30s; ETag-поллинг бесплатен, Р14).
- **Слои конфига фабрики**: дефолты в jar (`src/main/resources/application.yaml` самой
  фабрики — в целевом проекте этого файла НЕТ), оператор переопределяет env-переменными
  (bootstrap: env побеждает). `~/.gnomish/application.yaml` как операторский файл —
  design-опция, решить не позже loop (у loop настроек станет много, слот-политика Р3
  обязана быть явной). **Описать слои конфигурации в инструкции оператора / README.**
- **Токен трекера**: только окружение (имя переменной — design, напр.
  `GNOMISH_GITHUB_TOKEN`); из yaml не читается вовсе — секрет в файле проекта попал бы
  в git.
- **Сознательно НЕ конфиг v1** (ужесточение Р13 «оба — конфиг»: ручки по боли):
  критерий готовности = «открыт + ready-label» (имя label'а уже конфигурируемо — этого
  достаточно); порядок очереди = asc по номеру (даёт GitHub List); провижининг —
  всегда включён; формат маркеров/ack — свобода адаптера при round-trip (Р12).
- **Семантика «открыт»**: `state == "open"`; в фиде — серверный фильтр `state=open`,
  в `fetchTask` и на границе раунда — поле точечного GET (`closed` → скип Р7 / отзыв
  Р6). **PR-ловушка**: список issues возвращает и pull request'ы (у PR есть поле
  `pull_request`) — адаптер обязан их отфильтровывать, иначе `gnomish:ready` на PR
  заставит фабрику взять PR в работу; контрактный тест обязателен.
- **instanceId = `<instance-name | hostname>-<короткий случайный суффикс на процесс>`**
  (вариант-композит): атомарность lease на instanceId не держится (её решает comment
  id — Р13), id отвечает за «свой/чужой» (суффикс: уникальность при копипасте конфига
  и pid-reuse) и диагностику (префикс: чья машина; для публичных репо оператор задаёт
  нейтральное имя — ещё довод, что префикс в конфиге фабрики). Отклонены: hostname+pid
  (pid-reuse, контейнерные pid=1, утечка hostname в публичный issue), чистый random
  (нет диагностики), чистое имя из конфига (копипаста → два инстанса считают чужой
  claim своим — оба работают над одной задачей).
- **Форк-сценарий**: `.gnomish/` форка продолжает указывать на чужой `tracker.repo` —
  защита: (1) claim = запись, прав на чужой репо нет → 403; (2) стартовый провижининг
  label'ов (Р13) = smoke-тест привязки, смерть на старте с внятной ошибкой, не посреди
  задачи; (3) строка в инструкции оператора: «форкнул — поправь `tracker.repo`»
  (легитимность привязки на другой репо не даёт угадать намерение автоматически).

### Р21. Состав двух инструкций (принято 2026-07-20)

Обе — обязательные артефакты change'а (Р1/Р13); где физически живут (`docs/`? рядом с
шаблоном `.gnomish/`?) — design.

**Инструкция оператора**:
1. Быстрый старт: секция `tracker` в `.gnomish/config.yaml`, токен в окружении,
   слои конфигурации фабрики (jar → env; Р20 — обязательная секция);
2. Как отдать задачу: повесить ready-label; label'ы создаются сами с цветами
   (конфигурируемы, перекраска существующих — только в UI) и подсказками (Р13);
3. Словарь label'ов и кто их двигает: таблица Р4 в физике label'ов; человеку можно:
   ответ-комментарий + `needs-human` → `ready`, закрыть issue (отзыв); руками
   `working` не трогать;
4. Эскалации и решения: отчёт → ответ → ack «acting on decision»; гонка
   «консоль vs трекер» (Р8);
5. Поведение снимка (Р10): правки issue под взятой задачей не влияют; каналы влияния
   — решение или отзыв + новая задача;
6. Escape hatch: зависший Working мёртвого инстанса до heartbeat — ручной перевод
   label (Р7);
7. Projects v2 (Р15): доска — витрина/параллельная вселенная; auto-add по
   label-фильтру; reference-workflow моста «колонка → ready» (артефакт в составе
   change);
8. Форк: «поправь `tracker.repo`» (Р20);
9. CLI: `gnomish take` — формы, exit codes, headless «ответь в трекере и перезапусти».

**Инструкция автора стороннего адаптера** (сквозной пример — Redmine):
1. Модель: словарь состояний Р4 + матрица переходов; три уровня
   (состояние/исход/слот); адаптер отдаёт факты, политику применяет ядро (Р5);
2. Порт `Tracker`: операции Р12 с семантикой;
3. Контрактная спека — закон: одна на все адаптеры (testing.md), in-memory эталон —
   образец; ключевые свойства: фильтры `listReady` (включая аналог PR-ловушки:
   «отдавай только задачи»), наблюдаемая атомарность claim (эталон симулирует гонку),
   round-trip структурных маркеров;
4. Маппинг состояний на физику трекера: как сделано в GitHub (labels) и как могло бы
   быть в Redmine (статусы); свобода формата маркеров при обязательном round-trip;
5. Снимок, решения после ack, факты абортов — обязанности `fetchTask` /
   `collectDecisions`;
6. Известные ограничения и советы: sanitize-коллизия (Р16), экономика поллинга
   (ETag-аналоги, лимиты — Р14 как образец анализа).

### Р22. Конфиг-секция tracker: общее ядро + подсекция адаптера (принято 2026-07-20)

- **Принцип**: словарь конфигурации — часть адаптера, как и маппинг на физику трекера
  (продолжение Р1/Р13). Ядро знает единственный ключ-дискриминатор `type` и общие
  политики ядра; всё физическое (label'ы GitHub, статусы/переходы Jira, id Redmine) —
  типизированная подсекция, схему которой владеет и валидирует сам адаптер (fail-fast
  с агрегацией ошибок сохраняется, валидатор подсекции живёт в адаптере).
- **Эталонная структура** (`.gnomish/config.yaml`):

  ```yaml
  tracker:
    type: github            # дискриминатор — единственный ключ, который знает ядро
    abort-threshold: 3      # политика ядра (Р5) — общая для ЛЮБОГО трекера
    github:                 # словарь адаптера; схему владеет и валидирует адаптер
      repo: owner/repo
      labels:
        ready:       { name: "gnomish:ready",       color: "2ea44f" }
        working:     { name: "gnomish:working",     color: "1f6feb" }
        needs-human: { name: "gnomish:needs-human", color: "d73a4a" }
        delivered:   { name: "gnomish:delivered",   color: "8250df" }
  ```

- **Валидация на стыке**: `type: github` без секции `github` — ошибка; секция, не
  соответствующая `type` (лежит `jira:` при `type: github`) — тоже ошибка, не
  молчаливое игнорирование (опечатка/недопереключённый конфиг).
- **«Общая форма маппинга состояний для всех адаптеров» отклонена**: GitHub'у нужен
  цвет, Jira различает статусы и переходы, Redmine — числовые id; общий
  `tracker.states.*` стал бы наименьшим общим знаменателем с досыпанными
  adapter-specific ключами. Типизированная подсекция повторяет структуру кода:
  порт общий, адаптеры разные.
- **Дельты к принятому**: Р20 — `abort-threshold` на общем уровне секции (политика
  ядра, теперь видно и в структуре), `repo`/`labels` — внутри `tracker.github`;
  Р21 (инструкция автора адаптера) — новый пункт «адаптер объявляет и валидирует свою
  конфиг-подсекцию, общий уровень не трогает; in-memory эталон — минимальный случай
  (подсекция пустая/отсутствует)». Контрактная спека от конфига не зависит — она
  поверх порта.

### Р23. Корпоративный GitHub: `api-url` и каноника (принято 2026-07-20)

- **Принято: `api-url` — обязательный ключ `tracker.github`** (без дефолта в коде),
  даже для github.com (`api-url: https://api.github.com` — строчка приезжает из
  шаблона быстрого старта, Р21 п.1). Издания: github.com (`api.github.com`),
  Enterprise Cloud data residency (`api.<tenant>.ghe.com`), Enterprise Server
  (`https://host/api/v3` — другой хост И путь) — поэтому явный полный URL, а не
  «хост + угадывание пути». Доводы: прецедент Р3 (обязательная явность без скрытого
  дефолта), WireMock-тесты всё равно требуют перенацеливания на localhost (путь
  нестандартного URL всегда протоптан), ослабление «обязателен → опционален с
  дефолтом» обратносовместимо, обратное — ломающее.
- GHES-совместимость: наши API-выборы (list issues, labels, comments) — старые
  стабильные API, есть в любом GHES; ничего нового (issue types, sub-issues, advanced
  search) не используем (Р13/Р14). Лимиты на GHES настраиваются админом; ETag работает.
- **Каноника — принято (2026-07-20), вариант A** (выведен из ситуационной матрицы:
  поверхности id — ветка/worktree, task.json, структурные комментарии, round-trip
  адаптера, глаза оператора; ситуации — обычная жизнь, resume другим инстансом,
  смена привязки, переименование owner/repo, корпоративный GitHub, смена типа
  трекера, сторонний адаптер):
  - **формат `github:owner/repo#42`; хост включается ТОЛЬКО при не-дефолтном
    `api-url`** (`github:ghe.example.com/owner/repo#42`); условие — точное сравнение
    с константой `https://api.github.com` (нормализация мелочей URL — design);
  - **id самодостаточен** (координаты внутри): «короткий id + координаты из конфига»
    отклонён — конфиг становился бы частью идентичности, смена `tracker.repo` молча
    переинтерпретировала бы старые ветки на другой репозиторий;
  - **префикс `github:` остаётся**: пространства имён источников не пересекаются по
    построению (ad-hoc `manual-...` при существующем логине `manual`, будущий
    `jira:...`); каждый адаптер получает непересечение бесплатно; при смене типа
    трекера старые id опознаются как чужие с внятной ошибкой;
  - **канон — константа кода, НЕ конфиг**: принцип — настройка, смена которой задним
    числом ломает уже созданные артефакты (ветки, claim-маркеры), — не настройка, а
    мина; красота/краткость — слой отображения (`status`, короткие формы — косметика
    из Р16), он настраиваем безопасно;
  - отклонены: хост-всегда (вечный налог `github.com-` на массовый случай + скрытый
    special case «какой хост писать»), настраиваемый шаблон (мина), суррогат node_id
    (переживает переименования, но ветки нечитаемы, round-trip через GraphQL, нет
    аналога у других трекеров — упомянуть в инструкции адаптера как рассмотренный);
  - **пометка в design (переименование owner/repo)**: GitHub держит редиректы, включая
    API — вмороженные id продолжают работать; id = координаты на момент первого claim;
    проверку «каноника с чужим репо = ошибка» (Р19) делать с оглядкой на rename
    (предупреждение/сверка через редирект, не слепой отказ).
