## Что входит в factory loop

- **Автономный фид** (Р2): долгоживущий poll-цикл по `listReady`; пустая очередь →
  ждать интервал и опрашивать снова (webhook'ов нет — «no inbound HTTP»); интервал —
  конфиг. Однозадачный авто-режим («взять голову очереди и завершиться») уже в
  tracker-port — loop превращает его в непрерывный цикл.
- **Слоты и лимит параллелизма** (Р2): планировщик, N задач одновременно, «взял
  следующую, когда слот освободился».
- **Слот-политика hold/release целиком** (Р3): обязательная явная настройка;
  hold-поллинг («держатель следит, не вернул ли человек задачу»), release-паркинг.
  Осознанное следствие hold: все слоты могут повиснуть в ожидании людей.
- **Мультизадачный явный режим** (Р7): несколько id в запуске, скип невзятых с
  отчётом, итоговая сводка; агрегация exit-кодов — design-вопрос loop.
- **Heartbeat / stale-claim** (Р4, Р5, Р7): новые операции в порт `Tracker`
  (абстракция расширяема — Р12); снятие claim мёртвого инстанса; «кража» claim при
  явном указании id зависшей Working-задачи. До heartbeat ручной escape hatch
  оператора (перевод состояния руками) — уже задокументирован в tracker-port.
  Сюда же двойной отказ из Р5 (git и трекер мертвы → повисший claim).
- **Backoff-фильтр фида по фактам абортов** (Р5): задача в Ready с неистёкшим backoff
  невидима автономному фиду; форма (экспоненциальный с потолком) и параметры — конфиг.
  Факты (abortCount, lastAbortAt) порт отдаёт уже в tracker-port — здесь только
  политика ядра.
- **Политики очистки worktree по возрасту** (Р6): worktree отозванных/эскалированных
  задач копятся — чистка по возрасту.

## Фон, у которого дедлайн наступает с factory loop

- **Песочница command-чеков** (из explore-notes-future-changes.md): решение о
  песочнице нельзя откладывать дальше factory loop — автономный режим исполняет чужие
  команды без человека у консоли.

# Explore-сессия 2026-07-20: проектирование factory loop

Решено на входе:

- **Параллелизм сразу, без сужения до N=1** (решение оператора): один воркер —
  вырожденный случай параллельного запуска; планировщик для N=1 — тот же код с
  семафором на единицу, протокол (claim/heartbeat/abort/backoff) от N не зависит.
  Предпосылки уже готовы: реентерабельность движка (change 1), worktree-на-задачу
  (git-workflow), MDC-на-поток и interleaving-хуки in-memory эталона (tracker-port
  5.16, 2.6). Дефолт лимита параллелизма — скромный (2–3), конфиг.
- **Скелет**: тело слота = существующий take-цикл (claim → run → реакция на исход);
  новое в loop — только рамка: фид, планировщик, heartbeat, слот-политика, уборщик
  worktree, жизненный цикл процесса.
- **Что параллелизм добавляет в design** (не потерять): (1) git-физика — N слотов
  одновременно делают fetch/worktree в одном целевом репо, проверить блокировки
  общего клона; (2) планировщик не выдаёт одну задачу двум слотам — контрактный
  тест; (3) экономика записи: heartbeat × N задач против вторичных лимитов GitHub
  (500 записей/час, Р14).

Нити сессии: **А** планировщик и режимы запуска · **Б** heartbeat/stale-claim ·
**В** hold/release при N слотах · **Г** жизненный цикл процесса ·
**Д** песочница command-чеков · **Е** рамка change'а.
Порядок: по одной, начали с Б; решения и новые вопросы фиксируются здесь.

## Нить Б: heartbeat и stale-claim (РЕШЕНО 2026-07-20)

Решения; обоснования — в подсекции «Исследование лучших практик» ниже:

- **Б1. Физика heartbeat** (решено): beat = редактирование существующего
  claim-комментария, НЕ новые комментарии — тред issue остаётся читаемым (UX4),
  id комментария (якорь lease-паттерна Р13) не меняется. Одна запись на beat.
  В нагрузку beat'а — человекочитаемый прогресс («стадия X, попытка N, жив в
  HH:MM», паттерн Temporal): живой прогресс в треде issue бесплатно (NFR-O).
- **Б2. Протухание — по локальному наблюдению версий, часы не сравниваются**
  (решено; паттерн DynamoDB Lock Client): версия claim'а = (comment id,
  updated_at); протух = «версия не изменилась спустя TTL по монотонным часам
  наблюдателя с момента его первого наблюдения». Никакой арифметики
  now − updated_at. Новый инстанс не может объявить чужой claim протухшим раньше
  TTL от первого своего взгляда — льготный период по построению.
- **Б3. Конфиг — общий для всех инстансов** (решено): интервал beat'а + TTL
  множителем интервала (целое, ≥3; несогласованность невыразима) — в
  `.gnomish/config.yaml` секции `tracker` рядом с `abort-threshold` (принцип Р20:
  протокольные константы проекта; инстанс с коротким TTL крал бы у живого инстанса
  с редким beat'ом). Величины — Б8.
- **Б4. Обнаружение протухших** (решено): reaper — часть периодики фида каждого
  инстанса, роль санитарная: протухший Working возвращается в Ready, НЕ забирается
  себе — взятие обычным порядком очереди (возврат в оборот не зависит от свободных
  слотов нашедшего). Порт: операция листинга Working-задач с фактами-версиями
  claim'ов; память наблюдений и TTL-политика — в ядре (симметрия Р5); `listReady`
  не трогаем (контракт — только Ready). Beat'ятся только Working-задачи;
  held-задачи hold-политики — вопрос В (кандидат: не beat'ить, park-граница делает
  старый claim необструктивным).
- **Б5. Протокол перехвата** (решено): структурный маркер-граница «stale claim
  снят» (аудит смены держателя, аналог k8s leaseTransitions; якорь boundary для
  следующего verify-read) → удаление мёртвого claim-комментария (все инстансы под
  одним токеном — физически можно) → label working → ready. Гонка двух
  перехватчиков безопасна: обе операции идемпотентны по смыслу, дальше обычный
  lease по раннему comment id. Мандат явного режима (Р7): кража только протухшего
  Working; TTL-ожидания у оператора нет — ему показывают факты (держатель, возраст
  последнего beat'а как подсказка) и требуют явного подтверждения (TTY-диалог;
  headless — явный флаг); авторитет — человек, ошибку страхует fence Б6. Свежий
  Working без подтверждения — отказ с именем держателя.
- **Б6. Зомби-держатель** (решено): защита двухслойная — (1) проверка «claim ещё
  наш» на границе раунда (расширение проверки Р6); (2) git non-fast-forward push
  как fence (Kleppmann: ресурс сам отвергает опоздавшего): оба пишут в
  `gnomish/<task>` без force — второй получает отказ persist → штатный Aborted.
  **Инвариант безопасности: ветку задачи НИКОГДА не force-push'ат** — ни зомби,
  ни перехватчик при resume. Отдельная проверка перед git push не нужна (fence
  закрывает жёстче); но незафенсенные ЗАПИСИ В ТРЕКЕР (park/finish/release)
  предваряются дешёвой проверкой «claim ещё наш» (ETag) — остаточное TOCTOU-окно
  стоит мусорный label, не порчу данных, сходится записями нового держателя.
- **Б7. Отказ beat'а у живого инстанса** (решено): таксономия вместо счётчика M —
  сеть/5xx: WARN в лог, работать дальше (граница раунда решит: пройдёт, покажет
  кражу или сама упадёт → Aborted); 404 «комментария нет»: claim потерян
  (перехвачен/снят) — однозначный сигнал, на ближайшей границе реакция как на
  отзыв Р6 (остановиться, push best-effort — fence рассудит, слот освободить,
  трекерное состояние не писать — задача не наша).
- **Б8. Величины beat/TTL** (решено): первичная ручка — интервал beat'а,
  TTL = множитель × интервал (Б3). Ограничения: бюджет записи GitHub общий на
  токен (все инстансы оператора!), 500/час; латентность возврата задачи после
  смерти инстанса = TTL + до одного прохода фида; терпимость к сбоям сети =
  (множитель−1) × интервал. Принято: дефолт интервал 5 мин, множитель 3 (TTL
  15 мин) — 12 записей/час на работающую задачу, восстановление после смерти
  инстанса ~15–20 мин; обе ручки с дефолтами в `.gnomish/config.yaml`; джиттер
  beat'ов между слотами — деталь design.
- **Б9. Долгий раунд гнома (часы) при минутном TTL** (решено): beat — обязанность
  ИНСТАНСА, не гнома: отдельный heartbeat-поток инстанса редактирует claim'ы всех
  его Working-задач по расписанию, пока слот-поток заблокирован на исполнителе;
  гном о трекере не знает (Р11) — beat через него не проходит. Claim/beat отвечают
  на вопрос «жив ли держатель-процесс», НЕ «прогрессирует ли работа». Зависший гном
  при живом инстансе — другой отказ с другим механизмом: `RoundTimeout` агент-
  исполнителя (уже в коде, D7 add-agent-executor: по бюджету раунда процесс гнома
  убивается, раунд завершается штатно). Аналогия Temporal: heartbeat timeout vs
  start-to-close timeout — два разных таймаута; но в отличие от Temporal (там
  активность сама зовёт heartbeat) наш beat автоматический на уровне инстанса —
  потому и совместим с «гном = что угодно: скрипт, CLI, внешний сервис».
  Полезная нагрузка beat'а — из событий движка (EngineEventListener: стадия,
  попытка, «исполняется с»), не от гнома. Кража claim'а посреди долгого раунда
  обнаруживается 404-каналом Б7 за ≤интервал, реакция — на границе раунда (Р6,
  cancel посреди раунда отложен осознанно).
- **Б10. Таймаут работы гнома — постадийный, в руках оператора** (решение
  оператора, 2026-07-20): бюджет раунда (таймаут на работу гнома) должен
  конфигурироваться ОТДЕЛЬНО ДЛЯ КАЖДОЙ СТАДИИ — двухчасовая implement-стадия и
  минутная verify-стадия не могут жить с одним потолком. В design сверить с
  существующим `RoundTimeout` агент-исполнителя: где настройка живёт сейчас,
  постадийная ли она (Mechanism-секция стадии в `.gnomish/`?); если нет —
  дельта конфига в составе loop-change.

Вопрос, всплывший из Б в нить В (записан, обсудить там): физика возврата held-задачи
при hold-политике — человек ставит `ready`, фид ЧУЖОГО инстанса тоже это видит;
hold — преференция или монополия? Если держатель проигрывает гонку повторного
claim'а — задачу штатно подбирает чужой инстанс (resume из ветки), держатель
освобождает слот. Кандидат-решение: hold = преференция, не гарантия.

### Исследование лучших практик по heartbeat/stale-claim (2026-07-20)

Источники: Kleppmann «How to do distributed locking» (fencing-токены), Kubernetes
Leases (node heartbeat, leader election), AWS DynamoDB Lock Client, SQS visibility
timeout, Temporal activity heartbeats. Выводы в привязке к Б1–Б7:

- **Протухание — по локальному наблюдению, НЕ по сравнению часов** (паттерн
  DynamoDB Lock Client; правит Б2/Б4): наблюдатель запоминает «версию» claim'а
  (comment id + updated_at как версия) и момент наблюдения по СВОИМ монотонным
  часам; протух = «версия не изменилась спустя TTL по моим часам с момента моего
  первого наблюдения». Часы инстансов и сервера вообще не сравниваются; никакой
  арифметики now − updated_at. Следствия: (1) факт от адаптера — «версия claim'а»,
  память наблюдений и TTL-политика — в ядре (симметрия Р5 сохраняется);
  (2) свежезапущенный инстанс не может объявить чужой claim протухшим раньше, чем
  TTL от собственного первого наблюдения — корректный льготный период бесплатно.
- **Соотношение beat/TTL ≈ 1/3 — отраслевая норма** (k8s node lease 10s/40s,
  DynamoDB пример 3s/10s; подтверждает Б3): TTL = множитель ×3–4 от интервала.
  Наш масштаб времени иной (раунды минутами) → beat единицы минут, TTL десятки
  минут; цена TTL — латентность возврата задачи в оборот после смерти инстанса.
- **Fencing-токены обязательны — lease сам по себе зомби не останавливает**
  (Kleppmann; подтверждает Б6): ресурс обязан отвергать записи протухшего
  держателя. Наш fence — git non-fast-forward push (для двух не-force писателей
  эквивалентен монотонному токену). «Ветку задачи НИКОГДА не force-push'ат» —
  инвариант безопасности протокола, не вежливость. Незафенсенными остаются записи
  в ТРЕКЕР (park/finish/release — label и комментарии): перед ними — дешёвая
  проверка «claim ещё наш» (ETag-чтение); окно TOCTOU остаётся, но цена — мусорный
  label/комментарий, не порча данных; сходится следующей записью вора.
- **Бонус-канал обнаружения кражи** (следствие Б1 + перехват Б5): beat = PATCH
  своего claim-комментария; если reaper комментарий удалил — beat получает 404.
  Таксономия отказов beat'а (правит Б7): сеть/5xx → работать дальше (граница
  раунда решит); 404 «комментария нет» → claim потерян, финишировать раунд
  бессмысленно — реагировать как на отзыв на ближайшей границе.
- **Beat с полезной нагрузкой прогресса** (Temporal heartbeat details): раз beat
  и так редактирует claim-комментарий — включить в него человекочитаемый прогресс
  («стадия X, попытка N, жив в HH:MM») — прогресс в треде issue бесплатно (NFR-O);
  перенос состояния следующему держателю НЕ нужен — у нас состояние богаче, в ветке.
- **Подтверждения существующих решений**: SQS maxReceiveCount → DLQ ≡ наш K-фьюз →
  park(INFRA); SQS «terminate visibility timeout» ≡ наш release; at-least-once —
  дубль-доставка возможна при любом lease → работа обязана быть идемпотентной, у
  нас есть (раунды в ветке, resume); k8s leaseTransitions (счётчик смен держателя)
  ≡ наш маркер «stale claim снят» как аудит смены держателя — практика
  подтверждена.
- **В другие нити**: k8s v1.36 «release lease on exit» — graceful shutdown обязан
  явно release'ить claim'ы, не ждать TTL (→ нить Г); SQS-потолок продления 12h —
  нужен ли потолок удержания claim'а на held-задаче при hold-политике (→ нить В).

## Нить А: планировщик и режимы запуска (РЕШЕНО 2026-07-20)

- **А1. Одна машинерия — три режима** (решено): планировщик с N слотами и
  общим слот-телом (= take-цикл: claim → run → реакция на исход); режимы
  различаются источником задач × условием остановки: одиночный take (уже есть,
  без планировщика), пакетный явный (список ref'ов → до исчерпания списка →
  сводка), демон (фид `listReady` → бесконечно, до SIGTERM). Р2: лимит N
  действует в обоих новых режимах.
- **А2. Пакетный режим = varargs у `take`** (решено): `gnomish take 42 43 44` —
  то же намерение, что одиночное взятие; диспозиция по состояниям и скипы — Р7
  как есть. Exit-коды: один ref — таблица D16 без изменений, несколько —
  агрегат (А4).
- **А3. Имя команды демона — `serve`** (решено): симметрия `run` (ад-хок) /
  `take` (из трекера, с завершением) / `serve` (непрерывное обслуживание
  очереди); `loop` описывает реализацию, `work` конфликтует со статусом Working.
- **А4. Агрегация exit-кодов пакета** (решено, принцип): семейство «инструмент
  не смог» (<10) доминирует над семейством легитимных исходов (≥10); конкретика
  внутри семейств — design.
- **А5. Интерактивность — только у одиночного `take <ref>`** (решено): пакет и
  демон безоговорочно неинтерактивны — эскалация всегда park + отчёт в трекер,
  никаких TTY-диалогов (N задач на одну консоль); Р8-диалог остаётся привилегией
  одиночного take. Следствие для В: hold в демоне — чистый поллинг трекера.
- **А6. Механика фида** (решено): проход фида = poll `listReady` +
  backoff-фильтр ядра (уже в take 5.4) + reaper-обязанность (Б4); claim головы —
  в потоке фида, слоту передаётся уже захваченная задача (слоты одного инстанса
  не гоняются за одну голову — гонка остаётся только межинстансной, её решает
  lease); размещение конфига интервала опроса пустой очереди — ВЫНЕСЕНО в
  отдельное обсуждение по просьбе оператора (вернуться после В/Р3-дельты).
- **А7. Drain-режим — берём** (решено): флаг демона «пустая очередь = выход
  вместо сна» для cron-операторов без долгоживущего процесса (Р18 благословил
  cron-сценарий, голый take масштабируется там плохо); дёшев поверх той же
  машинерии; имя флага — design.

## Нить В: слот-политика → WIP-лимит (РЕШЕНО 2026-07-20; ПЕРЕСМАТРИВАЕТ Р3)

> **Пересмотр tracker-port Р3**: hold/release слот-политика ОТМЕНЕНА. Р3 вводил
> hold ради противодавления (лимит незавершённой работы) — эту функцию берёт на
> себя release-only + глобальный WIP-лимит W (В3). Оператор подтвердил: W
> закрывает ровно ту проблему, ради которой вводился hold. В propose loop-change
> — MODIFIED-дельта к решению Р3 с этим обоснованием.

- **В1. Сценарий-обоснование (оператор)**: большая готовая очередь (200+, в
  пределе 1М) + высокая доля эскалаций (криво заданные критерии, недостающие
  данные — реалистично, эскалация случается на ранней стадии, т.е. быстро) +
  release без ограничителя → фабрика = конвейер Ready → AwaitingHuman: за ночь
  сотни припаркованных задач, ветки от main на момент claim расходятся до
  несливаемости, входящая очередь человека необозрима, токены до точки эскалации
  сожжены. Настоящая функция hold в Р3 — противодавление (лимит незавершённой
  работы), а не локальность worktree.
- **В2. Почему не hold**: hold жёстко сцепляет лимит фронтов с числом слотов
  (W = N), держит исполнителей в простое и упирается в решения Б: held-claim
  надо либо beat'ить сутками (~850 записей за 3 суток ожидания — бюджет Б8),
  либо признать протухшим (монополии всё равно нет — park-граница Р13). Плюс
  рассинхрон с tracker-port 0.2 («эскалация всегда паркует и завершается»).
- **В3. Предложение — release-only слоты + глобальный WIP-лимит** (дельта к Р3,
  Kanban/закон Литтла: «перестань начинать, начни заканчивать»):
  - **N** — ёмкость исполнения инстанса (слоты), конфиг фабрики;
  - **W** — лимит открытых фронтов проекта (задачи в Working + AwaitingHuman
    суммарно, W ≥ N) — протокольная константа всех инстансов → `.gnomish/`
    секция tracker (рядом с abort-threshold, критерий Р20);
  - фид: возвращённые задачи ВНЕ лимита и В ПРИОРИТЕТЕ (их взятие закрывает
    фронт, не открывает; «сначала заканчивай»); свежие — только пока
    открытых < W;
  - лимит достигнут → новое не берём + явная наблюдаемость («X задач ждут
    решений, новые не беру» — давление на узкое звено, человека);
  - мандат явного take пробивает лимит (Р5/Р7);
  - лимит мягкий: гонка инстансов даёт перелёт на единицы — приемлемо для
    политики (не инвариант безопасности);
  - следствия release-only: беззатратно исчезают held-claim'ы, beat'ы
    припаркованных, атрибут held|parked, переходы 3 и 7 матрицы Р4 (3/4
    сливаются, 7 исчезает, остаётся 8), ручка слот-политики и вопрос «потолок
    удержания» (SQS 12h) — беспредметны;
  - закрепить взамен: переиспользование существующего worktree при resume тем
    же инстансом — проверить механику материализации (tracker-port 5.6) в design.
- **В4. Что нужно порту**: факт «число/список открытых задач» (GitHub: выборки
  по label working/needs-human, ETag) + признак «возвращённая» у ReadyTask
  (park-маркер в истории — факт адаптера) для приоритезации; ложится в ту же
  операцию листинга, что нужна reaper'у Б4. Сигнатуры — design.
- **В5. Граница метода** (в инструкцию оператора): WIP-лимит ограничивает ЧИСЛО
  одновременных веток, но не конфликты между ними — интеграционная дисциплина
  (rebase-стадия, нарезка задач) остаётся заботой автора пайплайна.
- **В6. Дефолт W** (открыто, к обсуждению): абсолютное число в `.gnomish/` (не
  выразить «×N» — N у инстансов разный); кандидат ~10–15.

## Нить А-доп: автомат фида и темп опроса пустой очереди (РЕШЕНО 2026-07-20)

Определения: **N** — слоты инстанса (конфиг фабрики); **W** — лимит открытых
фронтов проекта = count(Working)+count(AwaitingHuman) глобально (`.gnomish/`, В3);
«свежая» задача открывает фронт (можно при open<W), «resume» (человеком-возвращённая
или reaper'ом-освобождённая, ветка есть) закрывает посчитанный фронт — всегда можно,
в приоритете, вне лимита.

**Ключ упрощения: освобождение слота — ЛОКАЛЬНОЕ событие** (инстанс узнаёт без
трекера). Опрос трекера по таймеру нужен только когда слот свободен, а взять нечего.

Автомат фида инстанса:

| Состояние    | Условие                               | Разблокирует           | Опрос                          |
|--------------|---------------------------------------|------------------------|--------------------------------|
| Filling      | свободный слот И есть что взять        | —                      | опрос→claim→сразу снова, БЕЗ паузы |
| Idle-empty   | свободный слот, open<W, Ready пусто    | появилась Ready        | таймер холостого хода          |
| Idle-blocked | свободный слот, open≥W                 | человек вернул / reaper освободил фронт | таймер холостого хода |
| Full         | свободных слотов нет                    | локальное «слот свободен» | НЕ опрашивает, ждёт локальное событие |

Сквозные обязанности вне автомата: heartbeat своих Working (5 мин, записи, Б8);
reaper чужих Working (попутно на опросе `listReady`).

- **АД1. Reaper разблокирует и Idle-blocked** (тонкость): протухший claim мёртвого
  инстанса — Working-фронт, держит бюджет W, пока не подобран; reaper переводит его
  Working→Ready, open уменьшается — бюджет освобождается сам, на проходе фида, без
  человека. Значит Idle-blocked ждёт НЕ только человека → довод против «реже опрашивать
  в blocked».
- **АД2. Один интервал холостого хода** (решено): развод на два темпа (пусто vs
  W-достигнут) отклонён — оба опроса бесплатны по rate limit (условный GET → 304,
  Р14), реальная цена = пробуждение потока + строка лога (копейки), blocked не чисто
  «ждём человека» (АД1), два числа = лишняя ручка. Один интервал, **конфиг фабрики**
  (темп инстанса — разнобой между инстансами протокол не ломает: claim решает lease,
  W проверяется в момент взятия, reaper достаточно одного шустрого живого инстанса),
  дефолт **~30 с**, применяется только в Idle-*; в Filling паузы нет, в Full таймера
  нет. По духу Р20 «ручки по боли»: второй интервал — осознанная добавка потом, если
  Idle-blocked зашумит логи.

## Нить А-доп2: фид при нескольких инстансах (РЕШЕНО 2026-07-20)

Корень всего: все инстансы оператора ходят под ОДНИМ токеном → бюджет rate-limit
общий. Что общее (трекер): Ready-очередь, W, claim-комментарии, бюджет записи.
Что своё: слоты N, ETag, idle-таймер.

- **АМ1. Магнитуда интервала опроса от числа инстансов НЕ зависит** (решено):
  опрос = условный GET → 304, бюджет не тратит при любом M; ~30 с (АД2) остаются.
- **АМ2. «Стадо» на claim — настоящий многоинстансный эффект фида, лечится
  джиттером, не интервалом** (решено, намерение; детали — design): фид отдаёт
  очередь в порядке (Р13) → все голодные инстансы идут на ОДНУ голову → при M
  инстансах и 1 новой задаче M−1 впустую жгут claim-записи; на длинной очереди
  синхронное толкание даёт ~M² лишних попыток. Средства: (1) джиттер фазы опроса;
  (2) claim из головной ЗОНЫ (случайная из первых K готовых) — «старые первыми»
  мягкая преференция, лёгкое переупорядочивание приемлемо. Спасительный
  ограничитель: одновременных claim ≤ Σ свободных слотов, не длина очереди.
- **АМ3. Совокупный параллелизм ΣN упирается в общий бюджет записи через
  heartbeat** (решено, ключевой вывод; → инструкция оператора): heartbeat'ятся
  только Working (≤ ΣN); записей/час ≈ ΣN × (60/beat_мин); при beat 5 мин и резерве
  ~половины 500/час под claim/переходы/отчёты/reaper → **ΣN ≲ 20 одновременных
  задач на весь пул**. Регулятор потолка — интервал beat'а (Б8), не число
  инстансов: больше параллелизма → длиннее beat (медленнее возврат задач мёртвых
  инстансов). Б8 (beat) и ΣN связаны общим бюджетом записи — назвать явно в
  инструкции оператора.
- **АМ4. Прочее без проблем**: reaper дубль-безопасен (идемпотентен, АД/Б5);
  W-перелёт мягкий и растёт с M на единицы (В3); ETag/idle-таймер на инстанс, не
  делятся; честность claim (кто раньше запостил) при джиттере ~равномерна.

## Нить Г: жизненный цикл процесса демона (РЕШЕНО 2026-07-20, кроме помеченного design)

Долгоживущий `serve` вносит то, чего не было у одноразового take: сигналы остановки,
перезапуск, длительная недоступность трекера. Почти всё закрыто решениями Б.

- **Г1. Два разных выхода, не путать** (решено):
  - **Drain (А7)** — «выйди, когда работа кончится»: стоп-claim при пустой
    очереди, занятые слоты ДОВОДЯТ задачи до завершения (finish/park), выход когда
    все слоты пусты. Патичный.
  - **SIGTERM** — «выйди сейчас», ограничен grace-окном (k8s ~30 с, а раунд — часы,
    Б9): стоп-claim сразу, слоты стоп на границе раунда + release claim, mid-round
    задачи уходят в путь протухшего claim. Быстрый.
- **Г2. Graceful shutdown = drain слотов + ЯВНЫЙ release** (решено; закрывает
  флаг Б «k8s v1.36 release on exit»): при SIGTERM не ждём TTL — на ближайшей
  границе раунда (точка Р6, состояние уже закоммичено) release claim'ов, что
  успели в grace-окно → мгновенный возврат в оборот вместо +TTL.
- **Г3. Жёсткая смерть уже переживается — плата за решённый Б** (решено): grace
  истёк, гном в середине часового раунда (cancel посреди раунда отложен Р6) →
  SIGKILL → это «инстанс умер» из Б: claim протух по TTL, reaper вернул, resume из
  ветки, закоммиченные раунды целы. Нового механизма НЕ надо; graceful — лишь
  ускорение. **Design-хвостик**: гном = subprocess, при SIGKILL родителя может
  осиротеть и писать в ветку — корректность спасает git-fence (Б6), но утечку
  процесса закрыть kill process group при выходе (OS-деталь design).
- **Г4. Перезапуск инстанса — ничего особенного** (решено): фабрика stateless
  (state = трекер + ветка); прежние claim'ы стоят под СТАРЫМ instanceId (суффикс
  на процесс, tracker-port 1.3) — новый процесс не узнаёт их как свои и хорошо:
  протухнут и подберутся обычным путём (возможно, им же). Перезапуск = чистый
  старт. На старте — существующий провижининг label'ов (Р20) как smoke-тест
  привязки (недоступен репо → смерть на старте с внятной ошибкой).
- **Г5. Терпимость к простою трекера — даром из Б2** (решено): модель «протух =
  версия не менялась TTL по часам наблюдателя С МОМЕНТА наблюдения» → во время
  outage наблюдатель не читает версию, отсчёт TTL не идёт, после восстановления
  начинается заново. Тотальный отказ трекера НЕ порождает ложных краж (держатель
  успевает возобновить beat раньше свежего TTL). Спец-логики не надо: фид ретраит
  с backoff и оживает.
- **Г6. Гном, работающий во время outage — тонкость отчётности** (решено, v1
  политика): раунды крутить можно (git-fence страхует), но park/finish/recordAbort
  — записи в трекер, они не проходят. Исход durable в ветке (git-workflow персистит
  TaskOutcome/EscalationReport) → запись в трекер = сверка, переигрываемая. v1:
  инстанс держит слот и ретраит терминальную запись с backoff (исход в ветке цел);
  умер посреди ретраев → ветка всё знает, работает Г3/Б.
- **Г7. Требование к resume: «сверка при возобновлении»** (design-требование,
  выведено из Г6): reaper подобрал задачу, в ветке уже терминальный исход
  (Completed/Escalated) → resume ОБЯЗАН распознать и до-записать отложенный
  park/finish в трекер, НЕ переигрывать работу. Механизм общий (branch-outcome vs
  tracker-state), назвать в design resume-логики loop.

## Нить Д: песочница исполнения (РЕШЕНО 2026-07-20 — явная позиция v1)

Дедлайн решения (нельзя молча откладывать дальше loop). Грунт из кода:
`CommandProcessRunner` = `sh -c` в worktree с ПОЛНЫМ наследованием env;
`AgentProcessLauncher` = агент с шеллом на хосте; изоляции в production нет
(только E2E-контейнеры, testing.md); `Workspace` абстрагирует «где корень», не
«как изолировано исполнение».

- **Д1. Поправка масштаба** (решено): объект изоляции — ВСЁ исполнение в worktree
  (агент + чеки + команды), НЕ только command-чеки. Agent-cli executor запускает
  ИИ-агента с шеллом — свободы больше, чем у чека; песочница только чеков = театр.
  Переформулирует будущий change (заметки называли узко «песочница command-чеков»).
- **Д2. Позиция v1 — песочницы НЕТ, исполнение на хосте в worktree** (решено
  явно): защитимо через модель доверенного исполнения (Д3), не безрассудно.
- **Д3. Контроль доступа в автономном режиме = ворота готовности** (решено): «кто
  может повесить `gnomish:ready`» = «кто может исполнить код на хосте». GitHub
  пускает к label'ам только triage/write → на приватном командном репо ворота =
  команда (случайный автор issue `ready` не повесит). Требование в инструкцию
  оператора (Р21): НЕ подключать авто-`ready` из недоверенных источников
  (авто-ready публичных issue через мост Р15 = открыть исполнение посторонним).
- **Д4. Код гнома всё равно исполняется** (решено): сборка/тесты его результата —
  неотъемлемо от назначения фабрики; защита хоста от СОБСТВЕННОГО продукта — задача
  sandbox-change; v1 принимает. Envelope v1 = доверенный командный приватный репо
  (= «v1 только доверенные репо» из заметок, допустимо).
- **Д5. Дыра, закрываемая СЕЙЧАС** (решено, обязательная задача): `CommandProcess-
  Runner` наследует полный env → `GNOMISH_GITHUB_TOKEN` утекает в command-чеки
  (`./gradlew test`), нарушение Р11; tracker-port 5.17/D17 скрабит только
  agent-launcher. Применить тот же declared-scrub-list к запуску command-чеков.
  Кандидат в tracker-port (если успеем), иначе в loop. → сверить при финальной
  проверке, не забыть.
- **Д6. Шов будущего change** (решено, намерение): исполнение идёт через сырой
  `ProcessBuilder` в двух местах (агент + чек) → sandbox-change вводит шов изоляции
  исполнения, через который проходят оба (адаптер контейнера/VM), по образцу
  executor `api | agent-cli`; E2E-контейнеры (testing.md) = capability уже есть,
  повышение до production-режима = тот change.
- **Д7. Триггер постройки** (решено): реальная песочница нужна для репо с
  недоверенными источниками задач / pipeline-config (публичные + авто-labeling,
  мультитенант, SaaS). До тех пор поддерживаемый режим — доверенная команда,
  приватный репо. Решение держит loop в бюджете (→ Е).

### Д-доп: оценка размера будущего sandbox-change (скоупинг 2026-07-20)

> Полный скоупинг вынесен в `explore-notes-sandbox.md` (корень проекта) — сырьё
> для отдельного sandbox-change по триггеру Д7. Ниже — сжатая версия.

Поверхность исполнения (из кода): `AgentProcessLauncher` (ИИ-агент с шеллом) +
`CommandProcessRunner` (`sh -c` сборка/тесты). Защищать 4 поверхности: ФС (вне
worktree), секреты в env (но AI-ключ гному НУЖЕН), сеть (нельзя отрезать — нужны
AI-провайдер, реестры, git-remote → политика, не выключение), процесс/хост +
взаимная изоляция N слотов. Напряжение: гному нужна настоящая среда разработчика +
сеть по политике → «дать окружение в коробке», не «запретить всё».

Лесенка тиров (виден дешёвый высокоценный срез и полная стоимость):

- **Тир 0 — env-allowlist (дни; Д5 — подмножество)**: конструировать env исполнения
  из белого списка (только AI-ключ), не наследовать; для агента И чеков. Закрывает
  утечку секретов — самый дешёвый и ценный выигрыш. Кроссплатформенно, без инфры.
- **Тир 1 — OS-конфайнмент ФС (~неделя, только Linux)**: выделенный UID на задачу,
  worktree — единственный на запись, rlimits. Закрывает порчу хоста + ресурсы. Но
  Linux-only (macOS `sandbox-exec` устарел — локальная разработка мимо), сеть НЕ
  изолирует.
- **Тир 2 — контейнер на задачу (недели, кроссплатформенно) = «настоящая
  песочница»**: bind-mount worktree, сконструированный env, лимиты, очистка = снос.
  Затрагивает: (1) шов изоляции Д6 — оба адаптера через порт host|container, новый
  порт + рефакторинг; (2) жизненный цикл — `StreamJsonEventMapper`/`CommandOutcome`
  теперь работают ЧЕРЕЗ границу контейнера; (3) сохранить монополию push (Р11) —
  гном коммитит внутри, фабрика пушит СНАРУЖИ; (4) N контейнеров, очистка, отлов
  осиротевших (зеркало Г3). Фора: E2E уже в контейнерах (testing.md), Docker в
  пререквизитах ADR; латентность старта ~секунды на фоне часовых раундов — ничто.
- **Тир 3 — сетевая политика + образ инструментов (недели; тут открытость)**: два
  трудных разрастающихся вопроса — (а) egress default-deny + allowlist (AI,
  реестры, git-remote) через namespace/прокси, security-критично; (б) ЧЕЙ образ
  тулчейна — фабрика не знает тулчейн целевого проекта (оператор задаёт образ в
  конфиге? базовый + Dockerfile в `.gnomish/`?) — новая конфиг/docs-поверхность,
  дизайн-вопрос с глубиной.

**Итог размера**: «настоящая песочница» = Тир 2+3 ≈ масштаб GitHub-адаптера
(отдельный proposal, возможно два; несколько capability; недели; два открытых
трудных под-вопроса egress + образ). Дешёвый срез: Тир 0 (секреты, дни) → Тир 0+1
(секреты + ФС + ресурсы, ~неделя, Linux-only, без сети) — это «хардненинг», не
песочница, снимает 2 угрозы из 4 для доверенного envelope. Вывод подтверждает Д:
выносить из loop v1 правильно; Тир 0 (allowlist) стоит подтянуть рано (Д5).

## Нить Е: рамка change'а (РЕШЕНО 2026-07-20 — шов на 2 change'а)

Инвентарь loop v1: планировщик+N слотов (А1), режимы serve/batch/drain (А2/А3/А7),
автомат фида+reaper (А6/А-доп), heartbeat (Б1–Б10), reaper/stale-claim/fence/outage
(Б4–Б7,Г5–Г7), WIP-лимит W (В3/В4), жизненный цикл (Г1–Г7), уборщик worktree (Р6),
многоинстансные доводки (А-доп2), config-дельты. Три новые порт-операции ×
(порт+2 адаптера+контракт) + ядро + демон-CLI. **Заведомо >4 недель одним change**
(правило scope) → нужен шов через основной массив, не обрезка краёв.

- **Е1. Наблюдение-ключ**: heartbeat+reaper ОТДЕЛИМЫ от планировщика — они про
  «держать claim во времени», а это делает даже одиночный `take` (взял на 2ч, умер
  → застрявший claim, tracker-port закрыл лишь ручным escape-hatch). Значит это
  связный самостоятельный change, улучшающий существующую команду.
- **Е2. Шов = 2 change'а** (решено, ведущее предложение):
  - **Change 1 `add-claim-heartbeat`** (~2–3 нед): heartbeat (любой держащий claim
    прогон бьётся), протухание по наблюдению (Б2), reaper, перехват (Б5), зомби/fence
    (Б6), outage+reconcile-on-resume (Г5–Г7); порт += heartbeat-запись + листинг
    Working-с-версиями, оба адаптера + контракт; **потребитель — существующий
    `take`** (одиночный + голый-авто получают heartbeat и авто-восстановление
    застрявших claim'ов); закрывает footgun застрявшего claim, traceability сходится.
  - **Change 2 `add-factory-serve`** (~3–4 нед): планировщик+N слотов, автомат фида,
    WIP-лимит W, serve/batch/drain, жизненный цикл (SIGTERM/shutdown), уборщик
    worktree, доводки А-доп2 (джиттер, головная зона); порт += счётчик открытых
    фронтов + флаг «возвращённая», адаптеры+контракт; **потребляет Change 1**.
  - Порядок жёсткий: демон без heartbeat течёт claim'ами мёртвых инстансов
    (запрещено заметками) → Change 1 строго перед Change 2.
- **Е3. Шов НЕ предаёт решение о параллелизме** (важно): проходит не по «сначала
  один инстанс, потом параллельный», а по «удержание claim vs конкуренция+демон».
  Протокол Change 1 полностью многоинстансный с начала (наблюдение-протухание,
  гонки перехвата, fence); Change 2 добавляет планировщик (конкуренцию внутри
  инстанса) и демона. Вырожденного N=1 в дизайне нет.
- **Е4. Мелочь/отложенное**: Д5 (скраб env для command-чеков) — маленькое,
  независимое: tracker-port (если успеем) или Change 1. Песочница — отдельно по
  триггеру Д7 (`explore-notes-sandbox.md`).
- **Е5. Открыто для финальной сверки**: имена change'ов (`add-claim-heartbeat`/
  `add-factory-serve` — рабочие); точная граница порт-операций между Change 1 и 2
  (листинг Working может обслужить и reaper, и часть WIP-счёта — не задвоить);
  куда сядет reaper как потребитель в Change 1 (обязанность в take-старте? отдельная
  команда `reap`?).

## Финальная сверка решений (ВЫПОЛНЕНА 2026-07-20)

Сквозной проход по всем нитям А–Е, Р-решениям tracker-port и артефактам активного
change'а add-tracker-port (design D1–D17, tasks, спеки, код `RoundTimeout`).
Дальше — `/opsx:propose add-claim-heartbeat` (первый из двух change'ов Е2).

### С1. Проверки фактов — ответы

- **Б10 закрыт «уже так»**: `roundTimeout` живёт в `executor.settings` стадийного
  манифеста (`stage.yaml`, Mechanism-секция) — постадийный уже сегодня; валидируется
  `AgentSettingsValidator` (секунды или ISO-8601). Конфиг-дельта в loop не нужна,
  в design — только ссылка.
- **Д5 — окно было открыто, но НЕ используем**: задача tracker-port 5.17 не
  реализована; текст D17 и MODIFIED-дельта agent-executor спеки называют
  потребителем скраба только `AgentProcessLauncher`, `CommandProcessRunner` не
  упомянут. Решение оператора: начатую задачу не менять — расширение скраба на
  command-чеки едет в **Change 1** (см. С5.5).

### С2. Расхождения с tracker-port (реконсиляция 0.2) — исправить в propose

- **А5 устарел**: 0.2/5.8 удалили decision-wait диалог полностью («эскалация всегда
  паркует и завершается, одинаково с TTY и без»). Формулировка А5 «Р8-диалог
  остаётся привилегией одиночного take» неверна: интерактивности при эскалации нет
  НИ В ОДНОМ режиме. Единственная новая интерактивность loop-мира — подтверждение
  кражи claim (Б5); это pre-claim подтверждение, не in-run ожидание, с D12 не
  конфликтует — но в propose Change 1 отклонение от «identical with and without
  TTY» (headless-кража требует явный флаг) назвать явно.
- **В3 сузить**: 0.2 зафиксировал «no direct take of parked tasks» (take задачи в
  AwaitingHuman — отказ с указанием пути возврата, задача 5.9); пункта Р5/Р7
  «мандат берёт припаркованную предохранителем» больше нет. Мандат пробивает
  backoff-фильтр и W-лимит **только для Ready-задач**; припаркованные возвращает
  человек label-флипом. Кража Б5 не задета — она про Working.

### С3. Пробелы протокола reaper — закрыты решением С5.2

- **Reaper vs короткоживущий процесс (Change 1)**: протухание по Б2 требует
  наблюдателя, живущего дольше TTL — одноразовый cron-`take` наблюдает один раз и
  умирает. Авто-reaping в Change 1 работает только внутри долгого прогона (часовые
  раунды позволяют); для cron-only оператора ручной escape hatch остаётся до
  `serve`; явная кража Б5 работает сразу (авторитет — человек, TTL не нужен).
  Записать честно в proposal Change 1.
- **Reaper молчит в Full (serve)**: фид в Full не опрашивает трекер, а reaper был
  «попутно на опросе `listReady`» (А6/Б4) — при насыщении всех инстансов протухшие
  claim'ы не возвращались бы, вопреки Б4 («возврат не зависит от свободных слотов
  нашедшего»). **Поправка к Б4/А6**: reaper — обязанность heartbeat-потока
  инстанса (живёт, пока жив хоть один claim), не прохода фида; цена — одно
  ETag-чтение листинга Working на такт heartbeat'а.
- **АМ2 — явная дельта к Р2/Р13**: «claim из головной зоны» меняет «фабрика берёт
  с головы» (Р2) — в propose Change 2 оформить Р2-дельтой, чтобы контрактная
  формулировка не разошлась с реализацией.

### С4. Стыки, которые сходятся (проверено, вопросов нет)

- Вынос из А6 (интервал опроса пустой очереди) закрыт АД2; флаг «release on exit»
