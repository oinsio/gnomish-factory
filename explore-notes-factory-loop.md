## Что входит в factory loop

- **Автономный фид** (Р2): долгоживущий poll-цикл по `listReady`; пустая очередь →
  ждать интервал и опрашивать снова (webhook'ов нет — «no inbound HTTP»); интервал —
  конфиг. Однозадачный авто-режим («взять голову очереди и завершиться») уже в
  tracker-port — loop превращает его в непрерывный цикл.
- **Слоты и лимит параллелизма** (Р2): планировщик, N задач одновременно, «взял
  следующую, когда слот освободился».
- **Слот-политика hold/release целиком** (Р3): обязательная явная настройка;
  hold-поллинг («держатель следит, не вернул ли человек задачу»), release-паркинг.
  Осознанное следствие hold: все слоты могут повиснуть в ожидании людей.
- **Мультизадачный явный режим** (Р7): несколько id в запуске, скип невзятых с
  отчётом, итоговая сводка; агрегация exit-кодов — design-вопрос loop.
- **Heartbeat / stale-claim** (Р4, Р5, Р7): новые операции в порт `Tracker`
  (абстракция расширяема — Р12); снятие claim мёртвого инстанса; «кража» claim при
  явном указании id зависшей Working-задачи. До heartbeat ручной escape hatch
  оператора (перевод состояния руками) — уже задокументирован в tracker-port.
  Сюда же двойной отказ из Р5 (git и трекер мертвы → повисший claim).
- **Backoff-фильтр фида по фактам абортов** (Р5): задача в Ready с неистёкшим backoff
  невидима автономному фиду; форма (экспоненциальный с потолком) и параметры — конфиг.
  Факты (abortCount, lastAbortAt) порт отдаёт уже в tracker-port — здесь только
  политика ядра.
- **Политики очистки worktree по возрасту** (Р6): worktree отозванных/эскалированных
  задач копятся — чистка по возрасту.

## Фон, у которого дедлайн наступает с factory loop

- **Песочница command-чеков** (из explore-notes-future-changes.md): решение о
  песочнице нельзя откладывать дальше factory loop — автономный режим исполняет чужие
  команды без человека у консоли.
