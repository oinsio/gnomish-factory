# Сводка решений по change 2 и последующим changes

> Временный файл — всё, что обсуждено и решено в explore-сессии про изменения ПОСЛЕ `add-stage-engine`. Не для коммита. При propose соответствующего change — сырьё для proposal/design.

## Change 2: `add-manual-run` — ручной запуск движка

**Зачем:** первые НАСТОЯЩИЕ адаптеры к портам движка. Человек играет гнома — весь QC-цикл (verify, фидбек, попытки, эскалация, resume) виден вживую; заодно проверяется, не угаданы ли порты неправильно (порт-контрактные спеки уже готовы — интерактивные адаптеры обязаны их пройти). Плюс инструмент автора пайплайна: dry-run своего `.gnomish/` без AI.

**Решено:**
- **Однозадачный диалоговый режим** — единственный: `gnomish run --project <dir> --task "..."` (или `--task-file`); один процесс = одна задача = диалог = exit-код. Многозадачный shell (start/tasks/attach) — ОТКЛОНЁН (не усложнять). Скриптовый executor (гном = shell-скрипт) — решили не делать.
- **Интерактивные адаптеры** (человек играет роли):
  - executor: печатает цель задачи, `instructions.md`, входные артефакты, findings прошлых попыток и решения человека → «выполните стадию, Enter когда готово»; умеет вернуть DecisionNeeded (человек-гном просит решение);
  - external poll: «проверьте чек `<id>` сами: pass / fail / ещё идёт?»;
  - judge: печатает критерии из `acceptance.md`, спрашивает вердикт + findings текстом.
- **Реальные адаптеры**: `files_exist` (BuiltinCheckRunner, файловая система) и CommandCheckRunner (ProcessBuilder, exit-код = контракт). Command-раннер обязан класть хвост stdout/stderr в details финдинга — иначе фидбек гному пуст. Здесь же специфицируется wire-формат опционального findings-JSON от command-чеков (message/location/details — по доменной модели Finding).
- **Задача ad-hoc**, без трекера: описание из аргумента/файла, taskId генерирует раннер.
- **Workspace in-place**: каталог готовит человек руками (отдельный clone/worktree); раннер git не трогает.
- **Эскалация → resume в том же процессе**: отчёт в консоль → «введите решение или выйдите» → раннер сбрасывает attemptsUsed, дописывает Decision в контекст, зовёт run снова. (Resume между процессами невозможен до change 3 — персистентности нет.)
- **status**: команда в любом промпте диалога (`status`, `status --json`) + автосводки после каждой попытки / при эскалации / в финале. Рендер — чистая функция `(TaskContext, TaskState, активность из событий) → StatusReport → текст | JSON`. Формат секций: идентичность и позиция · текущая активность/pending · таблица попыток текущей стадии (чек, вердикт, findings кратко, время, токены) · суммарные метрики (время, токены, агрегат по инструментам) · последняя эскалация/решение. **JSON-контракт фиксируется здесь** — внешний CLI change 3 обязан выдавать идентичный JSON. Контракт живёт в **отдельной capability-спеке** (`status-report`), не внутри manual-run: `"version": 1`, camelCase, ISO-8601 UTC, длительности в мс; live-only поля (текущая активность) — опциональные, т.к. change 3 строит отчёт из state-файла без живого процесса; эквивалентность проверяется на границе попытки (точка покоя); якорь — reference-JSON (эталонный файл) в тестовых ресурсах. Поля `attempts[].result` и task-totals опираются на дельту, внесённую в change 1 (`AttemptRecord.result`, кумулятивные usage-итоги в `TaskState` — tasks 2.11/2.12); findings в JSON полные (усечение — забота текстового рендера); политика версии: аддитивно — без бампа, breaking — бамп.
### Черновик контракта status-report v1 (→ capability-спека `status-report` при propose)

Конвенции: camelCase; время — ISO-8601 UTC; длительности — мс; sealed-типы — дискриминатор `"type"` в lowerCamel; findings полные (усечение — забота текстового рендера); usage/токены опциональны (D5 движка); live-only поля — nullable (внешний CLI change 3 строит отчёт из state-файла, у него их нет). Пример (живой процесс, идёт verify второй попытки; ранее была эскалация с решением):

```json
{
  "version": 1,
  "task":     { "id": "manual-20260716-143502-x7", "title": "Fix flaky OrderServiceSpec" },
  "position": { "type": "atStage", "stage": "implement" },
  "activity": { "type": "verifying", "checkRef": "command:./gradlew test",
                "since": "2026-07-16T14:41:02Z" },
  "outcome":  null,
  "currentStage": {
    "attemptsUsed": 1,
    "attemptLimit": 3,
    "attempts": [
      { "round": 1,
        "result": "qualityFailure",
        "startedAt": "2026-07-16T14:35:10Z",
        "checks": [
          { "ref": "builtin:files_exist", "verdict": "pass", "findings": [], "durationMillis": 3 },
          { "ref": "command:./gradlew test", "verdict": "fail",
            "findings": [ { "message": "command exited with 1", "location": null,
                            "details": "…хвост stdout/stderr…" } ],
            "durationMillis": 41250 } ],
        "usage": { "wallMillis": 183000, "tokensIn": null, "tokensOut": null, "byTool": [] } } ]
  },
  "totals": { "wallMillis": 232000, "tokensIn": null, "tokensOut": null, "byTool": [] },
  "lastEscalation": { "type": "decisionNeeded", "stage": "plan", "at": "2026-07-16T14:20:44Z",
                      "question": "Refactor the retry helper or patch in place?",
                      "options": ["refactor", "patch"] },
  "lastDecision":   { "text": "patch in place", "author": "operator", "stage": "plan",
                      "at": "2026-07-16T14:21:30Z" }
}
```

Sealed-варианты: `position` = `atStage(stage)` | `pipelineEnd`; `activity` = `executing` | `verifying(checkRef)` | `awaitingInput(prompt)` | null; `outcome` = null (в процессе) | `completed` | `paused(passedStage)` | `escalated(report)` | `aborted(failedAt, cause)`; `attempts[].result` = `passed` | `qualityFailure` | `cannotVerify` | `decisionNeeded`; `lastEscalation.type` = пять видов `EscalationReport`. `totals` — кумулятивные по задаче (TaskState, задача 2.12 change 1), `attempts` — только текущая стадия. Якорь контракта — reference-JSON `status-report-v1.reference.json` в тестовых ресурсах change 2; тест change 3 сверяется с тем же эталонным файлом.

- **Логи**: file-appender с ротацией + MDC (`taskId/stage/attempt`) — ставит раннер на поток задачи; консольный/логирующий адаптер EngineEventListener даёт структурные строки. (Обязательство из обсуждения change 1: «логи обязательно прописать в change 2».)
- **Exit-коды по исходам** — два семейства с границей на десятке: `< 10` — «инструмент не смог нормально работать», `≥ 10` — «фабрика дошла до легитимного исхода» (скриптовому потребителю достаточно `$? -ge 10`); числа вне сигнальной зоны шелла (128+n), 1 и 2 — в общепринятых ролях:

  | Код | Значение                                                                               |
  |-----|----------------------------------------------------------------------------------------|
  | 0   | `Completed`                                                                            |
  | 1   | внутренняя ошибка (неожиданное исключение, недостижимый в change 2 `PipelineMismatch`) |
  | 2   | usage-ошибка (кривые флаги, неизвестный `--from-stage`)                                |
  | 3   | `.gnomish/` не загрузился (ошибки loader'а печатаются как есть)                        |
  | 4   | stdin закрылся посреди работы стадии (сценарий ввода исчерпан)                         |
  | 10  | `Escalated` — оператор вышел, оставив задачу в эскалации                               |
  | 11  | `Paused` — оператор вышел на checkpoint'е                                              |
  | 12  | `Aborted`                                                                              |

  Различение EOF по месту: Ctrl-D на промпте раннера (resume/checkpoint) = осознанный выход с кодом исхода (10/11); EOF внутри адаптера посреди стадии = код 4 (движок штатно вернёт Escalated(CannotVerify/CannotExecute), но раннер по флагу «ввод исчерпан» в DialogConsole минует resume-диалог) — для E2E код 4 означает «сценарий ввода не дописан», код 10 — «задача дошла до эскалации». Aborted: exhaustive switch, причина + сводка незаписанного состояния на stderr, spec с ломающимся фейком persist.
- **Spring-механика**: ApplicationRunner с разбором аргументов руками (без picocli); без аргументов — текущее bootstrap-поведение.

## Change 3: git-workflow (персистентность задачи)

- **Формат и версионирование state-файла**: сериализация TaskContext + TaskState (один файл или два — решить там; движку всё равно). **Обязательно включить и последний `TaskOutcome`/`EscalationReport`**: иначе `lastEscalation` (в т.ч. pending-вопрос `DecisionNeeded`) невосстановима из state-файла, внешний `status` не скажет, почему задача припаркована, и контрактный тест эквивалентности провалится.
- **Реальный AttemptPersistence**: коммит каждого раунда в ветку задачи (git как subprocess); семантика отказа уже готова в движке (Aborted). Файловая раскладка трейсов (например, `attempts/<stage>/<n>/trace.jsonl`) — решение этого change; связь с AttemptRecord — по ключу `(taskId, stage, attempt)`.
- **Worktree на задачу**: создание/управление рабочими копиями.
- **Resume между процессами/инстансами** из ветки — впервые по-настоящему.
- **Внешний CLI `gnomish status <task>`**: читает ветку (позже + трекер), живой инстанс не нужен. Контрактный тест: «StatusReport из событий ≡ StatusReport из state-файла той же задачи» — ловит расхождение моделей событий и персистентности.
- **Аналитика**: git-история state-файлов = архив всех попыток всех стадий (история в TaskState — только текущая стадия, старое читается из git).
- **Логи в git НЕ попадают** (решено при обсуждении change 2): лог — диагностика инстанса (процесс/машина, retention ~7 дней, `~/.gnomish/logs/`), не данные задачи; в ветку задачи — только структурные артефакты (state-файл, трейсы). Причины: для resume лог не нужен; в factory loop общий файл мешает строки N задач; git — навсегда и в чужом репо (ветка станет PR); вывод команд может нести секреты окружения.
- Строгая персистентность может перестать быть «просто адаптером порта» — решение «listener с особыми правами vs строгий порт» уже принято в пользу порта в change 1; здесь только реальная реализация.

## Change «factory loop» (цикл фабрики)

- **Tracker-порт + первый адаптер (GitHub)**; poll → claim → run движка → реакция на исход.
- **Claim-протокол с таймаутом/heartbeat** — обязателен независимо от Aborted: инстансы умирают молча (kill -9), задача должна возвращаться в очередь.
- **Протокол Aborted** («вежливая смерть»): ERROR в лог → best-effort комментарий в трекер (трекер ≠ git, обычно жив) → снять claim → ретрай через повторный подбор с backoff (НЕ внутри run) → предохранитель: K абортов подряд → Blocked с infra-отчётом. Счётчик абортов — в трекере (комментарии), т.к. git и отказал. K и backoff — конфиг цикла.
- **Эскалации**: Escalated → статус Blocked + отрендеренный отчёт. Для человека одна входящая очередь, два сорта отчётов: квалификационные (findings — нужен вердикт по задаче) и инфраструктурные (cannot verify / execute / persist — чинить окружение).
- **Resume-протокол**: сбор решений из комментариев трекера (спаривание с эскалационными отчётами — эвристика адаптера, отсюда опциональность полей Decision), сброс attemptsUsed, дописывание Decision, повторный run. Перемещение позиции — манипуляция состоянием, не команда в тексте.
- **Параллельные задачи**: N конкурентных run на виртуальных потоках, worktree на задачу, лимит в конфиге; MDC на поток. Движок готов (реентерабельность — требование change 1).
- **Прогресс в трекер** (NFR-O): короткие сводки со ссылками; подписчик — EngineEventListener.

## Прочие зафиксированные решения на будущее

- **Ревью — это QC-чек порождающей стадии, не отдельная стадия.** Cross-stage backtracking (возврат назад по пайплайну) — non-goal; если реальная практика потребует стадий-ревьюеров с откатом — отдельный change с семантикой «позиция назад + инвалидация артефактов промежуточных стадий».
- **Никакого inbound HTTP / статус-эндпоинта**: статус — через разделяемое состояние (ветка + трекер + логи). Пересмотр — только через ADR.
- **Бюджеты токенов/денег** — отложены (NG8 из load-pipeline-config, NG4 движка); шов — usage-поля AttemptRecord; для денег нужен прайс-каталог моделей.
- **Песочница для command-чеков** — отложенный риск (NFR-S): в ручном режиме команды исполняются в собственном репо человека — приемлемо; для автономного режима — решить до реального боевого запуска.
- **ai-provider порт** — приедет с реальными api-executor и judge-адаптерами.
- **Структурированные options у DecisionNeeded** (label + rationale вместо строк) — когда tracker-адаптер покажет, что строк мало.
- **Порядок changes**: add-stage-engine → add-manual-run → git-workflow → factory loop (порядок последних двух можно пересмотреть после change 2).
