# Бэклог explore: песочница исполнения (sandbox)

> Временный файл — скоупинг будущего sandbox-change, собранный в explore-сессии
> factory loop (2026-07-20, нить Д + Д-доп). Не для коммита. Сырьё для
> `/opsx:propose` будущего sandbox-change, когда наступит триггер (Д7).
> Родственные заметки: `explore-notes-factory-loop.md` (нить Д — позиция v1),
> `explore-notes-future-changes.md` (исходная пометка «песочница command-чеков»).

## Статус решения (из нити Д factory loop)

- **v1 factory loop — БЕЗ песочницы**, явная позиция: исполнение на хосте в
  worktree, как сейчас. Защитимо моделью доверенного исполнения (ниже), не
  безрассудно.
- **Envelope v1** = доверенный командный ПРИВАТНЫЙ репозиторий. Контроль доступа в
  автономном режиме — ворота готовности: «кто может повесить `gnomish:ready`» =
  «кто может исполнить код на хосте»; GitHub пускает к label'ам только triage/write
  → ворота = команда. Требование оператору: НЕ подключать авто-`ready` из
  недоверенных источников (авто-ready публичных issue через мост Р15 = открыть
  исполнение посторонним).
- **Код гнома всё равно исполняется** (сборка/тесты его результата) — неотъемлемо
  от назначения фабрики; защита хоста от собственного продукта фабрики — задача
  ЭТОГО change.
- **Триггер постройки (Д7)**: репо с недоверенными источниками задач/pipeline-config
  — публичные + авто-labeling, мультитенант, SaaS. До тех пор поддерживаемый режим —
  доверенная команда, приватный репо.

## Поправка масштаба (Д1): изолировать ВСЁ исполнение, не только чеки

Исходная заметка называла риском «песочницу command-чеков» — это узко. Объект
изоляции — всё исполнение в worktree: agent-cli executor запускает ИИ-агента с
шеллом (свободы больше, чем у чека), плюс command-чеки. Песочница только чеков =
театр. **Будущий change переименовать: «изоляция исполнения worktree».**

## Поверхность исполнения (из кода, 2026-07-20)

Два места на хосте исполняют код, на который влияет задача:

- `adapter/agent/AgentProcessLauncher` — запускает агент-CLI (напр. Claude Code)
  через `ProcessBuilder`, у агента шелл, он правит файлы и гоняет команды в worktree;
  stdout агента читает `StreamJsonEventMapper`.
- `adapter/check/CommandProcessRunner` — `sh -c <command>` в worktree с ПОЛНЫМ
  наследованием окружения родителя; результат — `CommandOutcome` (exit code +
  хвост stdout/stderr).

Изоляции в production нет. Есть только E2E-инфра (testing.md: Testcontainers +
sandbox-контейнеры для реальных agent-CLI прогонов) — capability контейнеризации
уже есть, повышение до production-режима исполнения = этот change. `Workspace` —
доменный порт, но абстрагирует «где корень», НЕ «как изолировано исполнение».

## Что защищать и от чего (threat model)

| Поверхность       | Угроза                                                        | Тонкость                                              |
|-------------------|--------------------------------------------------------------|-------------------------------------------------------|
| Файловая система  | чтение/запись вне worktree: хост, чужие worktree, `~/.gnomish`, ключи | у гнома должен остаться СВОЙ worktree на запись |
| Секреты в env     | токен трекера, push-креды, облачные ключи, API-ключи          | AI-ключ гному НУЖЕН — он им работает (agent-cli)       |
| Сеть              | эксфильтрация, обращения к чужой инфре, SSRF к внутренним      | нельзя отрезать: нужны AI-провайдер, реестры, git-remote → политика |
| Процесс/хост      | эскалация прав, fork-бомбы, забить диск, помешать другим слотам | N слотов одновременно → ещё и взаимная изоляция        |

**Главное напряжение**: гному нужна НАСТОЯЩАЯ среда разработчика (git,
компиляторы, пакетные менеджеры, AI CLI) + сеть по политике, а не выключенная. Это
«дать окружение в коробке», не «запретить всё» — потому и сложнее обычного sandbox.

**Существующие швы, которые песочница обязана сохранить**:
- монополия push (Р11 / git-workflow NFR-S1): гном коммитит, ФАБРИКА пушит — push
  идёт СНАРУЖИ песочницы, push-креды внутрь не попадают;
- AI-ключ — единственный секрет, легитимно живущий внутри (agent-cli режим).

## Лесенка тиров (дешёвый срез → полная стоимость)

- **Тир 0 — env-allowlist (дни, кроссплатформенно; Д5 — подмножество)**:
  конструировать окружение исполнения из БЕЛОГО списка (только AI-ключ), не
  наследовать родительское; для агента И для чеков. Закрывает утечку секретов —
  самый дешёвый и ценный выигрыш. Без инфраструктуры.
  - **Д5-задача (закрыть рано, кандидат в tracker-port/loop)**: `CommandProcess-
    Runner` наследует полный env → `GNOMISH_GITHUB_TOKEN` утекает в `./gradlew
    test` (нарушение Р11); tracker-port 5.17/D17 скрабит только agent-launcher.
    Применить тот же declared-scrub-list к запуску command-чеков. Тир 0 = обобщение
    этого до положительного allowlist.
- **Тир 1 — OS-конфайнмент ФС (~неделя, ТОЛЬКО Linux)**: выделенный
  низко-привилегированный UID на задачу, worktree — единственный путь на запись,
  rlimits (CPU/память/процессы/диск). Закрывает порчу хоста + исчерпание ресурсов.
  Ограничения: Linux-only (macOS `sandbox-exec` устарел — локальная разработка на
  darwin мимо), сеть НЕ изолирует.
- **Тир 2 — контейнер на задачу (недели, кроссплатформенно) = «настоящая
  песочница»**: каждый worktree исполняется в контейнере (Docker/Podman):
  bind-mount рабочей копии, сконструированный env, лимиты ресурсов, очистка = снос
  контейнера. Затрагивает:
  1. **шов изоляции (Д6)**: `AgentProcessLauncher` и `CommandProcessRunner`
     перестают звать сырой `ProcessBuilder`-на-хосте, идут через новый порт
     (`WorkspaceExecutor`/`Sandbox`?) с реализациями host-inplace | container —
     рефакторинг двух адаптеров + новый порт, по образцу executor `api | agent-cli`;
  2. **жизненный цикл контейнера**: создать, смонтировать, прокинуть
     stdout/stderr/exit code обратно — `StreamJsonEventMapper` (читает stdout
     агента) и `CommandOutcome` теперь работают ЧЕРЕЗ границу контейнера;
  3. **сохранить монополию push (Р11)**: гном коммитит ВНУТРИ, фабрика пушит
     СНАРУЖИ — шов не сломать;
  4. **параллелизм**: N контейнеров, очистка, отлов осиротевших (зеркало
     orphaned-subprocess Г3).
  - Фора: E2E уже в контейнерах (testing.md), Docker в пререквизитах ADR 0001;
    латентность старта контейнера ~секунды на фоне часовых раундов — ничтожна;
    bind-mount производительность на Mac — учесть.
- **Тир 3 — сетевая политика + образ инструментов (недели; ТУТ ОТКРЫТОСТЬ)**: два
  по-настоящему трудных, склонных разрастись под-вопроса:
  - **(а) egress-политика**: default-deny + allowlist (AI-провайдер, реестры
    пакетов, git-remote) через сетевой namespace/прокси/firewall. Security-критичная
    фиддли-часть, которую легко недоделать; самое ценное с точки зрения безопасности;
  - **(б) владение образом тулчейна**: контейнеру нужен тулчейн ЦЕЛЕВОГО проекта
    (JDK, gradle, node, …), а фабрика его НЕ знает. Варианты: оператор задаёт образ
    в конфиге (`tracker`/pipeline называет image); базовый образ + Dockerfile в
    `.gnomish/`. Новая конфиг- и docs-поверхность, дизайн-вопрос с глубиной.

## Итог размера

- **«Настоящая песочница» = Тир 2 + Тир 3 ≈ масштаб GitHub-адаптера**: отдельный
  proposal (возможно два — контейнеризация отдельно, сетевая политика/образ
  отдельно), несколько capability (шов исполнения, контейнерный адаптер, сетевая
  политика, конфиг образа, credential-allowlist), недели работы, два открытых
  трудных под-вопроса (egress, образ).
- **Дешёвый высокоценный срез**: Тир 0 (секреты, дни) → Тир 0+1 (секреты + ФС +
  ресурсы, ~неделя, Linux-only, без сети). Это «хардненинг», не песочница; снимает
  2 угрозы из 4 для доверенного envelope. Кроссплатформенность и изоляция СЕТИ
  начинаются только с Тира 2+3.
- **Вывод**: выносить песочницу из loop v1 правильно (Д). Тир 0 (env-allowlist)
  подтянуть рано вместе с Д5. Полноценный change заводить по триггеру Д7.

## Открытые под-вопросы будущего design

- egress-политика: механизм (сетевой namespace? прокси с allowlist? iptables?);
  как оператор задаёт allowlist доменов; как тестировать «deny сработал».
- владение образом тулчейна: конфиг image vs Dockerfile в `.gnomish/`; кто
  собирает/обновляет; версионирование; наследуется ли при форке (критерий Р20).
- имя и форма порта изоляции исполнения (host-inplace | container | future microVM).
- Тир 3+ (microVM Firecracker/gVisor/Kata) — для SaaS/мультитенанта, далёкое будущее.
- взаимодействие с уборщиком worktree (нить Е loop) — контейнер и worktree чистятся
  согласованно.
